# AI 智能体配置目录 (.agents)

本目录存储了所有面向 AI 智能体的详细规则、工作流、技能、脚本与知识资产。它是 `AGENTS.md`（全局路由）的具体实现落地，也是本项目智能体系统的核心组件容器。
本文件是 `.agents/` 目录定位、功能定位与职责展开的唯一主说明页；`AGENTS.md` 仅保留摘要与跳转。

## 目录定位
- **模块化边界**：`.agents/` 负责承载智能体相关资产，与业务代码、API 层、数据层、基础设施层保持清晰边界。
- **关注点分离**：将行为定义、执行规则、工作流、知识文档与自动化脚本集中组织，避免智能体逻辑分散在项目其他层。
- **可复用性**：目录内的规则、模板、技能与脚本优先按可迁移、可复用的方式设计，便于跨项目继承。
- **可测试性**：`.agents/` 为智能体能力提供稳定的测试边界，支持单元测试、集成测试与契约验证。

## 功能定位
以下定位描述的是 `.agents/` 在智能体系统中的职责承载方向；本节同时承接 `AGENTS.md` 中“1.2 功能定位”的目录级展开说明。

| 角色 | 职责描述 |
|------|----------|
| **技能管理中心** | 定义、注册和管理智能体可执行的技能、模板与工具资产。 |
| **状态管理载体** | 通过规范、知识文档、工作流上下文和相关产物，承载智能体运行过程中的状态约束与历史沉淀。 |
| **交互引擎支撑层** | 为用户输入处理、对话逻辑组织和响应生成提供规则、工作流与提示词层面的支撑。 |
| **配置中心** | 管理智能体的配置参数、行为规则、执行约束和路由策略。 |
| **决策控制器** | 沉淀智能体的决策逻辑、任务规划方法、执行流程与校验机制。 |

以下映射用于说明这些职责在 `.agents/` 内的主要落点与辅助落点。该映射是职责承载关系，而非严格的一对一模块切分：

### 技能管理中心

**作用**：定义、注册、组织和维护智能体可执行的技能、模板与工具资产。

**核心对象**：
- **技能定义**：技能说明文档、触发条件、输入输出约束与依赖信息。
- **模板资产**：标准化 `SKILL.md`、提示模板、参考示例与复用骨架。
- **评测产物**：触发评测集、回归样例、性能或兼容性验证脚本。

**实现模式**：
- 应采用统一目录约定管理技能资产，确保技能可发现、可迁移、可复用。
- 应将技能文档、脚本、评测和配套资源按同一技能边界聚合，降低维护成本。
- 应提供版本记录与最小契约校验，确保新增或变更技能具备稳定的可维护性。

### 状态管理

**作用**：维护智能体运行时状态，支持对话记忆和上下文管理。

**状态类型**：
- **对话状态**：当前对话历史、用户信息、上下文窗口。
- **长期记忆**：用户偏好、历史记录摘要、知识图谱。
- **运行时状态**：当前任务进度、执行状态、错误信息。

**实现模式**：
- 使用状态模式设计，避免不同阶段的上下文处理逻辑散落在各处。
- 应支持持久化承载，可根据场景使用内存、数据库或文件系统。
- 应提供状态快照与恢复能力，便于任务中断续跑、调试和回溯分析。

### 交互引擎支撑层

**作用**：为用户输入处理、上下文拼装、对话逻辑组织和响应生成提供统一的规则支撑。

**核心能力**：
- **输入理解**：识别用户意图、任务类型、约束条件与上下文依赖。
- **对话编排**：组织多轮对话流程、上下文注入、提示词拼装与阶段切换。
- **响应生成**：约束输出格式、解释粒度、引用方式与交互风格。

**实现模式**：
- 应采用规则、工作流与提示模板分层组合的方式，降低交互逻辑耦合。
- 应支持基于任务上下文的动态路由，使交互过程可复用、可扩展。
- 应提供可测试的对话策略单元，便于验证不同交互路径下的行为一致性。

### 配置中心

**作用**：统一管理智能体的配置参数、行为规则、执行约束和路由策略。

**配置类型**：
- **静态配置**：目录约定、文档边界、输出规范、技能模板等稳定规则。
- **任务配置**：工作流路由、工具选择、执行限制、输出样式等任务级参数。
- **环境配置**：路径约定、依赖初始化、外部工具可用性、特性开关等环境信息。

**实现模式**：
- 应采用分层配置结构，明确全局、项目、任务三个层次的继承与覆盖关系。
- 应优先使用声明式配置，减少隐式行为和分散硬编码。
- 应提供配置校验与默认值机制，降低配置漂移带来的执行风险。

### 决策控制器

**作用**：沉淀智能体的任务分解、路径选择、执行编排和结果校验逻辑。

**决策维度**：
- **任务判断**：识别任务类型、复杂度、依赖范围与优先级。
- **路径选择**：决定读取哪些规则、使用哪些工具、是否拆分步骤或并行执行。
- **结果校验**：在执行后进行自检、诊断、验证与必要的回退或补充说明。

**实现模式**：
- 应将决策逻辑设计为可审计的步骤，而非不可见的隐式跳转。
- 应支持“规划 -> 执行 -> 校验”闭环，确保任务推进过程可追踪。
- 应优先复用既有工作流与脚本能力，避免重复构建近似决策路径。

## 目录结构
- `rules/`：存放特定领域（如前端、后端、技能开发）的详细开发规范。
- `workflows/`：存放特定工作流（如 PR Review）的 AI 执行指南。
- `templates/`：存放供开发者和 AI 复用的标准化模板文件。
- `scripts/`：存放供智能体调用的自动化校验或执行脚本。
- `docs/`：专门为 AI 智能体保留的知识库、架构图和分析文档存储区。

## 🛠️ 技能标准化开发要求
为了统一管理所有的智能体技能（Skills），本目录已将 `SKILL.md` 约定纳入官方标准化要求：

1. **强制性约束**：在所有 `skills/` 目录下的子技能文件夹中，必须创建 `SKILL.md` 文件。
2. **文档规范**：详细要求请参阅 [技能开发规范 (rules/skills.md)](./rules/skills.md)。
3. **模板复用**：官方提供了一份标准的 [SKILL.md 模板](./templates/SKILL.md)，开发者在新增技能时应直接复制并按需修改。
