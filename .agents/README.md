# AI 智能体配置目录 (.agents)

本目录存储了所有面向 AI 智能体的详细规则、工作流、技能、脚本与知识资产。它是 `AGENTS.md`（全局路由）的具体实现落地，也是本项目智能体系统的核心组件容器。
本文件是 `.agents/` 目录定位、功能定位与职责展开的唯一主说明页；`AGENTS.md` 仅保留摘要与跳转。

## 目录定位
- **模块化边界**：`.agents/` 负责承载智能体相关资产，与业务代码、API 层、数据层、基础设施层保持清晰边界。
- **关注点分离**：将行为定义、执行规则、工作流、知识文档与自动化脚本集中组织，避免智能体逻辑分散在项目其他层。
- **可复用性**：目录内的规则、模板、技能与脚本优先按可迁移、可复用的方式设计，便于跨项目继承。
- **可测试性**：`.agents/` 为智能体能力提供稳定的测试边界，支持单元测试、集成测试与契约验证。

## 功能定位
以下定位描述的是 `.agents/` 在智能体系统中的职责承载方向；本节同时承接 `AGENTS.md` 中“1.2 功能定位”的目录级展开说明。

| 角色 | 职责描述 |
|------|----------|
| **技能管理中心** | 定义、注册和管理智能体可执行的技能、模板与工具资产。 |
| **状态管理载体** | 通过规范、知识文档、工作流上下文和相关产物，承载智能体运行过程中的状态约束与历史沉淀。 |
| **交互引擎支撑层** | 为用户输入处理、对话逻辑组织和响应生成提供规则、工作流与提示词层面的支撑。 |

### 技能管理中心

**作用**：定义、注册、组织和维护智能体可执行的技能、模板与工具资产。

**核心对象**：
- **技能定义**：技能说明文档、触发条件、输入输出约束与依赖信息。
- **模板资产**：标准化 `SKILL.md`、提示模板、参考示例与复用骨架。
- **评测产物**：触发评测集、回归样例、性能或兼容性验证脚本。

**实现模式**：
- 应采用统一目录约定管理技能资产，确保技能可发现、可迁移、可复用。
- 应将技能文档、脚本、评测和配套资源按同一技能边界聚合，降低维护成本。
- 应提供版本记录与最小契约校验，确保新增或变更技能具备稳定的可维护性。

### 状态管理

**作用**：维护智能体运行时状态，支持对话记忆和上下文管理。

**状态类型**：
- **对话状态**：当前对话历史、用户信息、上下文窗口。
- **长期记忆**：用户偏好、历史记录摘要、知识图谱。
- **运行时状态**：当前任务进度、执行状态、错误信息。

**实现模式**：
- 使用状态模式设计，避免不同阶段的上下文处理逻辑散落在各处。
- 应支持持久化承载，可根据场景使用内存、数据库或文件系统。
- 应提供状态快照与恢复能力，便于任务中断续跑、调试和回溯分析。

### 交互引擎支撑层

**作用**：为用户输入处理、上下文拼装、对话逻辑组织和响应生成提供统一的规则支撑。

**核心能力**：
- **输入理解**：识别用户意图、任务类型、约束条件与上下文依赖。
- **对话编排**：组织多轮对话流程、上下文注入、提示词拼装与阶段切换。
- **响应生成**：约束输出格式、解释粒度、引用方式与交互风格。

**实现模式**：
- 应采用规则、工作流与提示模板分层组合的方式，降低交互逻辑耦合。
- 应支持基于任务上下文的动态路由，使交互过程可复用、可扩展。
- 应提供可测试的对话策略单元，便于验证不同交互路径下的行为一致性。

### 配置中心

**作用**：统一管理智能体的配置参数、行为规则、执行约束和路由策略。

**配置类型**：
- **静态配置**：目录约定、文档边界、输出规范、技能模板等稳定规则。
- **任务配置**：工作流路由、工具选择、执行限制、输出样式等任务级参数。
- **环境配置**：路径约定、依赖初始化、外部工具可用性、特性开关等环境信息。

**实现模式**：
- 应采用分层配置结构，明确全局、项目、任务三个层次的继承与覆盖关系。
- 应优先使用声明式配置，减少隐式行为和分散硬编码。
- 应提供配置校验与默认值机制，降低配置漂移带来的执行风险。

### 决策控制器

**作用**：沉淀智能体的任务分解、路径选择、执行编排和结果校验逻辑。

**决策维度**：
- **任务判断**：识别任务类型、复杂度、依赖范围与优先级。
- **路径选择**：决定读取哪些规则、使用哪些工具、是否拆分步骤或并行执行。
- **结果校验**：在执行后进行自检、诊断、验证与必要的回退或补充说明。

**实现模式**：
- 应将决策逻辑设计为可审计的步骤，而非不可见的隐式跳转。
- 应支持“规划 -> 执行 -> 校验”闭环，确保任务推进过程可追踪。
- 应优先复用既有工作流与脚本能力，避免重复构建近似决策路径。

## 对可维护性、可扩展性和模块化的影响

本节说明将智能体相关资产集中组织到 `.agents/` 目录后，对系统演进质量带来的结构性影响。

### 可维护性

**正面影响**：
- **清晰的边界**：规则、工作流、技能、脚本与知识资产职责明确，便于快速定位问题来源。
- **集中管理**：智能体相关内容集中在同一目录中，降低查找成本与跨目录追踪成本。
- **标准化结构**：统一的目录约定和资产组织方式降低学习成本，便于新成员快速建立全局认知。

### 可扩展性

**正面影响**：
- **技能扩展**：新增技能通常只需在 `skills/` 下增加独立目录和配套资产，无需侵入核心说明结构。
- **配置扩展**：可通过规则、模板或配置文件调整行为，减少为单一变化修改核心流程代码的需要。
- **状态扩展**：可通过新增状态约束、上下文文档或状态承载方式扩展系统能力，而不必重写现有模块。

**扩展模式**：
- **插件化架构**：技能以插件或独立能力单元形式注册，新增能力时尽量不修改已有核心路径。
- **配置驱动**：优先通过配置和规则驱动行为调整，而不是将差异直接硬编码进执行逻辑。
- **依赖注入**：通过接口约定、模板替换或脚本编排，灵活替换组件实现，降低耦合升级成本。

### 模块化设计

**模块化原则**：
- **高内聚**：相关功能集中在同一模块内，避免单一职责被拆散到多个目录。
- **低耦合**：模块间通过清晰接口、规则约定和目录边界交互，减少直接依赖。
- **可替换性**：模块可独立替换、迁移或重构，而不要求其他模块同步大范围改动。

**模块化收益**：
- 便于单元测试与局部验证，可独立校验规则、脚本、工作流或技能模块。
- 支持并行开发，不同角色或团队可围绕各自模块协作推进。
- 降低技术债务，使模块能够按自身节奏独立演进，而不持续放大系统级耦合。

## 最佳实践建议

本节沉淀 `.agents/` 目录在实际演进中的通用实践，帮助目录结构、技能实现、配置治理与状态管理保持长期稳定。

### 目录组织原则

1. **保持简洁**：优先控制高频开发路径的目录层级，通常建议不超过 3 层；对于 `docs/superpowers/` 这类归档型目录，可基于主题或资产类型适度分层。
2. **命名规范**：统一使用小写命名，并与仓库现有风格保持一致，优先采用连字符命名，如 `version-tracking.md`、`pr-review.md`。
3. **单一职责**：每个文件或目录应只承载一个明确的功能领域，避免将规则、脚本、模板和知识文档混放。

### 技能开发最佳实践

1. **技能粒度**：每个技能应聚焦单一能力边界，避免将多个弱相关能力打包成“大而全”技能。
2. **参数校验**：在技能执行前校验输入参数、依赖条件和上下文约束，尽量将失败前移。
3. **错误处理**：提供清晰的错误信息、恢复建议和必要的回退机制，降低排查成本。
4. **文档说明**：为每个技能提供完整的 `SKILL.md` 说明，覆盖描述、参数、依赖、部署、错误处理与版本记录。

### 配置管理最佳实践

1. **配置分离**：将敏感配置与代码分离，优先通过环境变量或受控配置文件注入。
2. **配置继承**：支持基础配置与环境覆盖的分层结构，避免复制整份配置文件。
3. **配置验证**：在启动或执行前验证配置的完整性、合法性和默认值是否生效。

### 状态管理最佳实践

1. **状态分层**：区分短期对话状态、长期记忆和运行时状态，避免不同生命周期的数据混杂。
2. **状态清理**：定期清理过期状态、无效缓存和临时产物，避免内存或上下文持续膨胀。
3. **状态序列化**：支持状态的序列化、反序列化与必要的快照恢复能力，便于中断续跑、调试和回溯。

## 常见反模式

本节归纳 `.agents/` 目录演进中应主动规避的高频问题。这些反模式会直接削弱目录边界、配置治理、模块解耦和长期维护能力。

### 需要避免的做法

| 反模式 | 问题描述 | 推荐做法 |
|--------|----------|----------|
| **单一大文件** | 将规则、流程、脚本或说明文档堆叠在单个文件中，导致职责混杂、难以维护。 | 按职责拆分到对应目录，如 `rules/`、`workflows/`、`scripts/`、`docs/`。 |
| **硬编码配置** | 在脚本或逻辑中直接写死路径、参数、密钥或环境差异，导致迁移和复用困难。 | 使用环境变量、受控配置文件和分层配置机制管理。 |
| **紧耦合** | 模块间直接依赖具体实现或隐式约定，导致局部修改牵动全局。 | 通过清晰接口、规则约定和可替换编排方式解耦。 |
| **缺少测试** | 技能、脚本或核心逻辑缺乏最小验证，容易在变更后引入回归。 | 为关键模块补充单元测试、集成校验或契约验证。 |
| **过度嵌套** | 高频开发路径的目录层级过深，增加定位成本和认知负担。 | 控制常用目录层级，一般不超过 3 层；归档型目录按需例外。 |

## 目录结构
- `roles/`：存放协作元模型中的 Role 实例，用于定义职责模板、默认规则绑定与权限边界。
- `rules/`：存放特定领域（如前端、后端、技能开发）的详细开发规范。
- `workflows/`：存放特定工作流（如 PR Review）的 AI 执行指南。
- `templates/`：存放供开发者和 AI 复用的标准化模板文件。
- `scripts/`：存放供智能体调用的自动化校验或执行脚本。
- `docs/`：专门为 AI 智能体保留的知识库、架构图和分析文档存储区。
