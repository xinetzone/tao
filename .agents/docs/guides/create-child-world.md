# 子世界创建实操指南

本文档面向 AI 智能体和人类开发者，说明如何在项目内创建一个符合继承规范的子世界。

处理多世界层级相关任务前，应先阅读 [`.agents/rules/world-hierarchy.md`](../../rules/world-hierarchy.md)。

---

## 1. 前置条件

当且仅当以下任一条件成立时，才需要创建子世界：

- 该目录下的代码/文档有**独立的规则需求**（如测试域需要专门的测试规范）。
- 需要**覆盖父世界的部分规则**（如子项目锁定不同的 Python 版本）。
- 需要**追加专属技能或工作流**，且这些能力仅在该目录范围内生效。
- 该目录是一个**独立子包或子模块**，Agent 在其中工作时需要额外的上下文约束。

> 如果目录内容完全遵循全局规则，无需额外约束，则**不要**创建子世界。保持层级扁平是推荐做法。

---

## 2. 创建步骤

### 步骤 1：在目标目录创建 `AGENTS.md`

在需要成为子世界的目录（如 `tests/`、`apps/my-service/`）下新建 `AGENTS.md`。

```bash
# 示例：为 tests/ 目录创建子世界
touch tests/AGENTS.md
```

### 步骤 2：声明继承关系

在 `AGENTS.md` 开头明确声明父世界引用，并说明冲突时的优先级。

必须包含以下要素：
- 引用父目录的 `AGENTS.md`（通常使用相对路径 `../AGENTS.md`）。
- 声明"本文件仅定义补充规则，不重复维护全局规则"。
- 声明冲突时以父世界为准（除非显式覆盖且已声明）。

### 步骤 3：添加「规则增量说明」节

在 `AGENTS.md` 中说明本子世界是**增量补充**还是**有覆盖行为**：

- **Fragment 扩展（无覆盖）**：仅新增父世界未定义的规则文件，同名规则文件不存在。
- **有覆盖**：存在与父世界 `.agents/rules/` 下同名的规则文件，或 `AGENTS.md` 中覆盖了父世界的同名节。

如有覆盖，必须进入步骤 5 编写「覆盖说明」。

### 步骤 4：（可选）创建 `.agents/rules/` 放置增量规则

若子世界需要新增或覆盖规则文件，创建对应目录：

```bash
mkdir -p tests/.agents/rules
```

将增量规则文件放入该目录。同名文件将完整覆盖父世界对应文件；非同名文件视为新增。

### 步骤 5：（可选）创建 `.agents/skills/` 或 `.agents/workflows/` 追加能力

若子世界需要专属技能或工作流：

```bash
mkdir -p tests/.agents/skills
mkdir -p tests/.agents/workflows
```

- `skills/` 下的技能**追加**到父世界技能集，不替换。
- `workflows/` 下的工作流**追加**到父世界工作流集，不替换。

## 3. AGENTS.md 模板

以下模板可直接复制使用，替换占位符即可。

```markdown
# 📁 <目录名> 智能体契约

这是 `<目录名>/` 目录的局部智能体契约，用于补充根目录 [`../AGENTS.md`](../AGENTS.md) 的全局规则。

## 1. 继承关系

- 本目录下所有任务必须优先遵循根目录 [`../AGENTS.md`](../AGENTS.md)。
- 本文件仅定义 <目录名> 域补充规则，不重复维护全局规则。
- 如本文件与根目录契约发生冲突，以根目录契约为准（除非已在本文件「覆盖说明」中显式声明）。

## 2. 规则增量说明

本子世界为 Fragment 扩展（增量补充），无覆盖父世界规则行为：

| 新增/覆盖规则文件 | 说明 |
|---|---|
| `.agents/rules/<规则名>.md` | <该规则的简要说明> |

> 如果存在覆盖行为，请替换上表为「覆盖说明」节，详见下文「覆盖说明节示例」。

## 3. <目录名> 域规则入口

处理 `<目录名>/` 目录下的 <具体任务类型> 任务前，必须先阅读：

- [`./.agents/rules/<规则名>.md`](./.agents/rules/<规则名>.md)

## 4. 目录职责

- `<目录名>/` 用于 <该目录的用途说明>。
- `<目录名>/.agents/` 仅用于存放 <目录名> 域 AI 规则与说明，不承载根级技能、工作流或长期知识库。
- <其他职责约束，如路径可移植性、禁止写入绝对路径等>。
```

## 4. `.agents/` 目录结构模板

子世界推荐的目录结构：

```
<子世界根目录>/
├── AGENTS.md                 # 子世界契约（必需）
└── .agents/
    ├── rules/                # 增量/覆盖规则（可选）
    │   └── <规则名>.md
    ├── skills/               # 追加技能（可选）
    │   └── <技能名>.md
    └── workflows/            # 追加工作流（可选）
        └── <工作流名>.md
```

> 子世界**禁止**包含 `world.toml`。一个 monorepo 仅允许一份 `world.toml`，位于项目根目录。

## 5. 覆盖说明节示例

当子世界需要覆盖父世界的规则时，在 `AGENTS.md` 中添加如下章节：

```markdown
## 覆盖说明

| 父规则文件 | 覆盖节 | 覆盖原因 |
|---|---|---|
| `python.md` | `## 3. 版本约束` | 子项目锁定 Python 3.11，与父世界 3.13 要求不同 |
```

覆盖粒度说明（详见 [`../../rules/world-hierarchy.md`](../../rules/world-hierarchy.md)）：

| 覆盖对象 | 最小单位 | 覆盖方式 |
|---|---|---|
| `.agents/rules/` 下的规则文件 | 文件（按文件名匹配） | 子世界同名文件完整覆盖父世界对应文件 |
| 规则文件内部的规则节 | Section（`##` 级标题） | 子世界同节声明覆盖父世界同节内容 |
| 上下文路由表 | 表格行（按 key 匹配） | 子表项追加或覆盖父表同名项，未提及项透传 |

> 未声明的隐式覆盖在语法上是合法的，但在审计时应标记为待审项。建议始终显式声明覆盖意图。

---

## 6. 禁止事项

以下行为在任何子世界中均不允许：

| 禁止项 | 原因 | 后果 |
|---|---|---|
| 创建独立的 `world.toml` | 一个 monorepo 仅允许一份世界级 manifest | 破坏世界发现机制，导致层级解析错误 |
| 覆盖 `world.toml` 中 `[kernel]` 声明的条目 | Kernel 为宇宙法则，任何子世界均不可修改 | Agent 必须显式报告冲突并拒绝执行 |
| 嵌套深度超过 3 层 | 对齐嵌套深度与觉醒量表推论 | 层级过深导致规则追溯困难，维护成本剧增 |
| 隐式覆盖不声明 | 破坏协作可审计性 | 其他 Agent 无法感知规则变化，产生意外行为 |

## 7. 常见问题

### Q1：何时需要子世界，何时不需要？

**需要**：目录有独立规则、需要覆盖父规则、需要追加专属技能/工作流。

**不需要**：目录内容完全遵循全局规则，没有额外约束需求。保持扁平，不要过度分层。

### Q2：子世界可以覆盖父世界的 `AGENTS.md` 内容吗？

可以，但仅限于同名 Section（`##` 级标题）的显式覆盖。建议通过「覆盖说明」节声明意图。根级契约的核心继承条款（如"以父世界为准"）不建议覆盖。

### Q3：如何验证子世界的合规性？

1. 检查是否存在独立的 `world.toml` —— 应不存在。
2. 检查 `.agents/rules/` 中的文件是否均为必要增量 —— 非同名文件为新增，同名文件为覆盖。
3. 如有覆盖，检查 `AGENTS.md` 中是否包含「覆盖说明」节。
4. 检查嵌套深度 —— 从根 `AGENTS.md` 到本子世界的层级数应 **≤ 3**。
5. 检查是否未触及 Kernel 条目 —— 任何对 `[kernel]` 的修改尝试均无效。

### Q4：子世界的 `.agents/skills/` 会替换父世界的技能吗？

不会。`.agents/skills/` 和 `.agents/workflows/` 的行为是**追加**而非替换。子世界技能与父世界技能合并后共同生效。

### Q5：子世界可以引用父世界的规则文件吗？

可以，且推荐这样做。子世界 `AGENTS.md` 中应使用相对路径引用父世界 `AGENTS.md`，并在上下文中明确继承关系。子世界未覆盖的规则自动透传，无需重复声明。
