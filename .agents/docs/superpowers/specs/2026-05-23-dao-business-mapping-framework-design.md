# Dao Business Mapping Framework Design

## Goal

为 `AgentForge` 建立一份人机共用的全链路框架设计稿，用于把马王堆帛书版《道德经》的哲学内核，稳定转化为可复用的产品能力、技术模块、验证指标与业务场景表达方式。

本次设计目标如下：

- 在现有哲学总纲基础上，补齐可执行的转化框架
- 同时服务人类决策与 AI 规划、写 spec、拆任务、做复盘
- 形成从战略、产品、技术到验证的统一表达语言
- 以少量代表性场景证明框架具备真实落地能力
- 保持首版结构简洁，避免沦为抽象口号或过度工程化模板

## Background

当前仓库已经具备以下基础：

- `AGENTS.md` 已将“极致简约、大道至简”与“反者道之动，弱者道之用”纳入项目级契约
- `.agents/docs/references/dao-tech-foundation.md` 已沉淀哲学概念、工程映射与常见失配问题
- `.agents/docs/superpowers/specs/` 已作为设计稿归档位置，用于沉淀可执行方案

这些基础足以说明项目已经完成“哲学立意”的建立，但仍缺少一份中间层文档，去回答以下问题：

- 哲学原则如何被翻译成产品能力，而不是停留在理念表述
- 产品能力如何进一步分解为模块、工作流、技能或规则
- 如何验证一条映射链路是否真实成立，而不是概念自洽
- 如何让人类与 AI 在同一份文档中共享判断标准与执行入口

因此，需要新增一份执行层设计稿，承接哲学总纲，并为后续 spec、计划、任务拆解与复盘提供统一骨架。

## Scope

- 定义一套从哲学原则到业务场景的全链路映射框架
- 明确人机共用的文档结构与字段模板
- 设计标准化的场景矩阵卡片
- 定义验证机制与复盘回流方式
- 提供 2-3 个代表性示例场景，验证框架可用性

## Non-Goals

- 本次不直接实现具体业务功能或代码模块
- 本次不穷举全部潜在行业或业务方向
- 本次不编写独立的自动化生成脚本、索引器或数据库结构
- 本次不把示例场景扩展成完整产品 PRD
- 本次不替代现有哲学总纲文档，而是作为其执行层补充

## Design Principles

1. 哲学驱动但不空泛：每条哲学命题都必须能落到工程解释和业务收益。
2. 极简优先：优先选择最小必要字段、最短转化路径和最低维护成本的文档结构。
3. 人机共读：人类应能快速理解，AI 应能稳定抽取并复用字段。
4. 回转校正：框架必须支持“设计 -> 实施 -> 验证 -> 复盘 -> 再抽象”的闭环。
5. 柔性落地：优先支持低侵入、可组合、可渐进扩展的实施路径。

## Positioning

这份设计稿位于现有哲学总纲与未来具体实施方案之间，承担“执行层母版”的角色。

其职责分工如下：

- `AGENTS.md`：定义最高优先级的项目级哲学与行为约束
- `references/dao-tech-foundation.md`：解释哲学内核、概念映射与常见失配问题
- 本设计稿：定义统一的全链路转化框架、矩阵模板、验证机制与示例场景
- 后续各类 spec / plan / retrospective：基于本设计稿展开具体任务与结果沉淀

## Options Considered

### Option A: 总纲优先

结构：

- 先写哲学原则与方法论
- 再在末尾追加少量场景

优点：

- 顶层逻辑完整
- 适合作为长期理念母版

缺点：

- 执行入口较弱
- AI 需要再次抽取场景字段，复用成本较高

### Option B: 矩阵优先

结构：

- 直接以大表形式组织 `哲学原则 -> 产品能力 -> 技术模块 -> 验证指标 -> 业务场景`

优点：

- 检索效率高
- 更接近执行与任务拆解

缺点：

- 容易损失哲学层解释
- 长期演进中容易碎片化

### Option C: 双层框架

结构：

- 上层总纲：解释转化原则与框架运行逻辑
- 下层矩阵：提供标准场景模板、验证清单与示例场景

优点：

- 同时满足长期母版与执行入口双重需求
- 最符合人机共用与全链路表达目标

缺点：

- 需要控制文档长度与信息密度

## Recommendation

采用 Option C。

推荐原因如下：

- 能把哲学层解释与执行层矩阵稳定分开，避免互相稀释
- 能同时满足“你自己做顶层判断”和“AI 直接拿来规划与拆解任务”的双重需求
- 与现有 `.agents/docs/references/` 和 `.agents/docs/superpowers/specs/` 的分层设计完全兼容

## Framework Overview

核心映射链定义为 5 层：

1. `哲学原则`
2. `产品能力`
3. `技术模块`
4. `验证指标`
5. `业务场景`

对应含义如下：

- `哲学原则`：定义价值取向和设计判断准绳
- `产品能力`：把哲学思想转成用户或业务可感知的能力单元
- `技术模块`：把能力继续下沉为模块、工作流、技能、规则或数据结构
- `验证指标`：规定如何判断这条链路是否真实成立
- `业务场景`：承载真实问题空间，验证前四层是否形成闭环

建议用以下 Mermaid 主图表达框架闭环：

```mermaid
flowchart LR
    A["哲学原则"] --> B["产品能力"]
    B --> C["技术模块"]
    C --> D["验证指标"]
    D --> E["业务场景"]
    E --> F["复盘反馈"]
    F --> A
```

## Transformation Logic

这套框架不是静态知识表，而是可回转的运行结构：

1. 从哲学原则抽象出能力方向
2. 将能力方向落成可实施的技术模块
3. 为模块与能力定义可观察的验证指标
4. 在具体业务场景中验证落地效果
5. 将复盘结果反向作用到前面的抽象与设计中

这一定义直接呼应“反者道之动”，即框架必须具备回看、纠偏和再次抽象的能力，而非一次性线性推进。

## Standard Mapping Unit

每个映射单元建议采用统一字段：

- `哲学命题`
- `工程解释`
- `对应能力`
- `技术落点`
- `验证方法`
- `适用场景`

这样设计的目的有两点：

- 对人类读者：快速理解一条完整因果链
- 对 AI：稳定抽取结构化字段，用于生成后续 spec、计划和任务分解

## Scenario Matrix Design

场景矩阵是整份设计稿的核心执行单元。后续每个业务方向都应尽量按照同一套字段描述，而不是以散文形式临时阐释。

建议每个场景使用以下字段：

- `场景名称`
- `场景目标`
- `哲学依据`
- `工程解释`
- `产品能力`
- `技术模块`
- `验证指标`
- `实施优先级`
- `风险与偏差`
- `复盘入口`

推荐模板如下：

```md
### 场景：<名称>

- 场景目标：
- 哲学依据：
- 工程解释：
- 产品能力：
- 技术模块：
- 验证指标：
- 实施优先级：
- 风险与偏差：
- 复盘入口：
```

首版建议采用“上层总览表 + 下层标准场景卡片”的组合，而不是只使用一张大表。原因如下：

- 人类读者更适合先理解场景卡片，再进入字段细节
- AI 仍然可以从固定字段中稳定抽取信息
- 后续扩展时，可以先新增卡片，再决定是否汇总成更大的矩阵导航页

## Validation Model

每个场景至少应通过以下三层验证：

- `哲学一致性`：是否真实体现了“大道至简”“反者道之动”“弱者道之用”
- `工程有效性`：是否降低复杂度、提升可维护性或增强回转能力
- `业务成立性`：是否在真实场景中减少摩擦、提升成功率或降低成本

建议每个场景都附带统一验证清单：

- 是否绑定了明确的哲学依据，而非泛化口号
- 是否抽象出了独立、可感知的产品能力
- 是否映射到清晰的技术模块，而非职责混杂
- 是否定义了可观察的验证指标
- 是否提前识别了风险与偏差路径
- 是否预留了复盘和纠偏入口

## Scenario Selection Strategy

首版只放 2-3 个示例场景，且应优先选择差异较大的方向，以验证框架具备跨层复用能力。

选择原则：

1. `代表性`：能够代表项目未来核心研究与落地方向
2. `异质性`：场景类型彼此不同，避免只是模板复写
3. `可落地性`：能够自然映射到仓库现有或可预期的模块、工作流、技能或知识资产

## Example Scenarios

### 场景：智能体柔性协作编排

- 场景目标：设计一套低侵入、可组合的 agent 协作方式，降低单一大流程的耦合度。
- 哲学依据：`弱者道之用`
- 工程解释：优先使用柔性接口、声明式规则和可替换编排，而不是刚性绑定的大一统执行流。
- 产品能力：按任务类型组合技能、工作流和知识入口，支持渐进扩展。
- 技术模块：`.agents/workflows/`、`.agents/rules/`、`.agents/skills/`
- 验证指标：新增场景接入所需改动范围小；不同能力能按边界独立演进；失败时可局部回退。
- 实施优先级：首批
- 风险与偏差：为了“统一”反而把不同能力重新耦合进单一流程。
- 复盘入口：相关任务完成后归档至 `.agents/docs/superpowers/retrospectives/`

### 场景：知识闭环与复盘驱动演进

- 场景目标：建立“设计 -> 执行 -> 复盘 -> 再抽象”的知识循环，避免任务经验一次性流失。
- 哲学依据：`反者道之动`
- 工程解释：在流程中预留回看、纠偏和再次抽象的机制，将复盘视为系统演进的一部分。
- 产品能力：让设计文档、执行计划与复盘文档形成连续链路，支持长期学习。
- 技术模块：`.trae/` 工作台、`.agents/docs/superpowers/specs/`、`.agents/docs/superpowers/retrospectives/`
- 验证指标：任务结束后能快速追溯设计意图；复盘结果能反向更新长期规范；重复问题出现频率下降。
- 实施优先级：首批
- 风险与偏差：复盘沦为形式化总结，无法反馈到规则和设计层。
- 复盘入口：以任务复盘文档为主，并根据需要同步更新 `AGENTS.md` 或 `.agents/docs/`

### 场景：业务能力最小闭环建模

- 场景目标：面对一个新的业务问题时，优先抽象出最小可行能力闭环，而非预先设计复杂平台。
- 哲学依据：`大道至简`
- 工程解释：将复杂目标拆回最小必要能力、最短调用链和最清晰职责边界，优先形成可验证的最小闭环。
- 产品能力：快速抽象业务问题、识别首批能力单元、明确优先级和落地路径。
- 技术模块：后续具体模块待按业务方向展开，可先落在 spec、plan 与能力拆解流程中。
- 验证指标：首版能力集合小而完整；没有为未确认需求引入多余抽象；实施路径明确且可分阶段推进。
- 实施优先级：首批
- 风险与偏差：把“简化”误解为“省略关键约束”，导致后续返工。
- 复盘入口：围绕对应业务方向的后续 spec 与任务总结进行回看

## Rollout Plan

建议按以下顺序落地：

1. 先将本设计稿归档到 `.agents/docs/superpowers/specs/`
2. 后续在需要时基于本设计稿生成更具体的实施计划
3. 从上述 3 个示例场景中择优展开为独立 spec 或任务计划
4. 在任务完成后，将复盘结果反向补充到长期规范与参考文档中

## Risks

- 如果场景字段定义过多，首版会增加维护成本并削弱极简原则
- 如果只保留哲学表述而缺少验证指标，框架会退化为概念宣言
- 如果示例场景彼此过于相似，无法证明框架具备跨场景复用价值
- 如果复盘结果不回流到规范层，闭环会中断

## Acceptance Criteria

- 新增设计稿位于 `.agents/docs/superpowers/specs/` 下
- 文档明确区分哲学总纲层与执行层
- 至少定义 1 套统一映射链与 1 套场景模板
- 至少定义 1 组统一验证清单
- 至少提供 2-3 个代表性示例场景
- 文档可同时被人类和 AI 直接读取并用于后续任务展开
