# Trigger Evals Review Notes

本文档为 `trigger-evals-final.json` 提供人工判断依据，帮助在 description optimization 或人工抽查时快速理解每条 query 为什么应该触发或不应触发 `task-execution-summary`。

## Review Criteria

- **应触发**：用户核心目标是对一个已完成或进入收尾阶段的任务做总结、复盘、回顾、经验沉淀、过程整理或 postmortem。
- **不应触发**：用户虽然提到了“完成”“事故”“Sprint”“学习”等相邻概念，但主目标是其他事情，例如写 README、做根因分析、排优先级、写外部说明、做代码审查、拆任务模板、出 PPT、给结论或记流水日志。

## Should Trigger

1. **登录权限开发复盘**
   - 结果：应触发
   - 原因：任务已完成，用户明确要求“复盘”，并希望提炼“下次怎么少走弯路”，属于典型开发任务总结。

2. **Sprint 收尾回顾**
   - 结果：应触发
   - 原因：用户要求“整理一份回顾”，且关注进度偏差和协作问题，属于典型项目管理复盘。

3. **线上报警排查记录**
   - 结果：应触发
   - 原因：故障已处理完毕，目标是“把排查过程和解决方案记下来”，重点是过程沉淀而非继续排障。

4. **POC 技术研究 recap**
   - 结果：应触发
   - 原因：出现 `recap`、`技术研究总结`、`哪些判断是对的` 等强信号，明确要求回顾已完成研究过程。

5. **Docker / CI 学习总结**
   - 结果：应触发
   - 原因：用户明确说“别讲新知识了”，改为总结学习过程、卡点和后续建议，属于学习复盘。

6. **Bug 关单后的过程整理**
   - 结果：应触发
   - 原因：Bug 已经关单，用户要整理“做了什么”“为什么判断错”“怎么定位出来”，这是标准的修复复盘。

7. **面向老板的结构化总结**
   - 结果：应触发
   - 原因：虽然输出将用于汇报，但用户当前请求是“给我一版结构化总结”，总结本身是主要产物。

8. **两周项目收尾复盘**
   - 结果：应触发
   - 原因：用户显式询问“做得怎么样”“有什么收获”“下次怎么做”，完全符合技能的隐式触发条件。

9. **事故 postmortem**
   - 结果：应触发
   - 原因：出现 `postmortem` 和“过程总结”“留档”等直接信号，是本技能的高置信触发样本。

10. **算法实验收尾总结**
   - 结果：应触发
   - 原因：实验已结束，用户要记录失败尝试、关键判断和后续行动项，属于研究/实验复盘。

## Should Not Trigger

11. **模块完成后写 README**
   - 结果：不应触发
   - 原因：虽然任务已完成，但当前目标是文档交接，不是总结过程或经验。

12. **事故后做根因分析与修复方案**
   - 结果：不应触发
   - 原因：用户明确说“不用再整理过程文档”，主目标是分析与修复，不是回顾。

13. **Sprint 后排下个迭代优先级**
   - 结果：不应触发
   - 原因：这是计划规划任务，关注未来排序，不是对已执行过程做复盘。

14. **学习后制定下一阶段路线**
   - 结果：不应触发
   - 原因：用户显式说“不用回顾之前过程”，目标是产出学习计划而非学习总结。

15. **故障记录转外部说明**
   - 结果：不应触发
   - 原因：虽然来源于排查记录，但当前产物是面向客户的外部沟通说明，而不是内部经验沉淀。

16. **修完 Bug 后做代码审查**
   - 结果：不应触发
   - 原因：用户明确要求检查回归风险和缺失测试，主要任务是 review，不是总结。

17. **根据重构经历拆 Jira 模板**
   - 结果：不应触发
   - 原因：这里是模板生成任务，虽可参考经历，但目标不是形成回顾报告。

18. **故障结束后提炼时间线做 PPT**
   - 结果：不应触发
   - 原因：用户明确说“不是要复盘”，目标是短汇报材料，不需要总结分析。

19. **POC 后直接要 go/no-go 结论**
   - 结果：不应触发
   - 原因：虽然源于已完成实验，但当前任务是决策建议，结论优先于过程总结。

20. **将当日工作记入 daily note**
   - 结果：不应触发
   - 原因：这里只是时间顺序日志记录，不要求分析得失、经验或行动项，不属于结构化任务复盘。

## Use Notes

- 如果后续真实触发测试中发现某些负样本被误触发，优先检查 description 是否对“完成了”“事故结束了”“Sprint 结束了”这类阶段性结束信号赋权过高。
- 如果正样本出现漏触发，优先补强以下语义：
  - “整理一下刚才做的事”
  - “做得怎么样 / 有什么收获 / 下次怎么做”
  - “记下排查过程 / 留档 / postmortem / recap”
