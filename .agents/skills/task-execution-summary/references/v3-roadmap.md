
## 2. 模块 B1: 自定义模板支持

### 2.1 功能概述与价值主张

**功能描述**

自定义模板支持模块允许用户和企业根据自身需求创建、管理和应用个性化的报告生成模板，突破现有固定格式的限制，实现报告输出的高度定制化。

**核心价值**

| 维度        | 当前痛点            | V3 解决方案           | 量化收益          |
| --------- | --------------- | ----------------- | ------------- |
| **品牌一致性** | 不同团队使用不同格式，难以统一 | 企业可定制符合 CI 的模板    | 品牌合规性提升至 100% |
| **场景适配**  | 固定模板无法满足特殊行业需求  | 支持法律/医疗/金融等行业定制模板 | 适用场景扩展 500%+  |
| **效率提升**  | 手动调整格式耗时        | 一键应用预设模板          | 格式调整时间减少 95%  |
| **合规要求**  | 监管报告有特定格式要求     | 模板内置合规字段和校验规则     | 合规风险降低 80%    |

**目标用户画像**

| 用户类型        | 典型需求           | 使用频率 | 定制深度         |
| ----------- | -------------- | ---- | ------------ |
| **个人开发者**   | 简单样式调整         | 低    | 浅层（颜色/字体）    |
| **技术团队负责人** | 统一团队报告规范       | 中    | 中层（章节结构/字段）  |
| **企业知识管理员** | 符合 ISO/GDPR 标准 | 高    | 深层（完整流程/审批）  |
| **系统集成商**   | 为客户定制解决方案      | 极高   | 完全定制（API/钩子） |

### 2.2 支持的模板格式及优先级

**格式支持矩阵**

| 格式           | 优先级     | 支持程度 | 适用场景              | 编辑复杂度 | 渲染性能      |
| ------------ | ------- | ---- | ----------------- | ----- | --------- |
| **HTML**     | P0 (必须) | 完整支持 | Web 展示、邮件嵌入、浏览器打印 | 低     | 高 (即时)    |
| **Markdown** | P0 (必须) | 完整支持 | Git 文档、静态站点、开发者工具 | 极低    | 高 (即时)    |
| **DOCX**     | P1 (重要) | 完整支持 | Word 文档编辑、正式报告提交  | 中     | 中 (需转换)   |
| **PDF**      | P1 (重要) | 完整支持 | 归档、打印、法律文档        | 高     | 低 (需渲染引擎) |

### 2.4 模板上传流程

#### 六步详细上传流程

```
Step 1: 准备阶段
├── 选择模板格式 (HTML/MD/DOCX/PDF)
├── 准备模板文件 (<= 5MB)
├── 定义元数据 (名称/描述/分类/标签)
└── 确认变量占位符语法正确

Step 2: 校验阶段
├── 格式验证 (文件类型/大小/编码)
├── 语法检查 (标签闭合/变量引用)
├── 安全扫描 (XSS/注入检测)
└── 变量映射验证 (必填变量是否齐全)

Step 3: 解析阶段
├── 提取变量列表
├── 分析模板结构 (章节/区块)
├── 识别条件逻辑 (if/for/switch)
└── 生成模板抽象树 (AST)

Step 4: 预览阶段
├── 使用示例数据渲染
├── 多设备预览 (桌面/移动/打印)
├── 变量高亮显示
└── 导出预览结果

Step 5: 发布阶段
├── 分配唯一模板 ID
├── 设置访问权限 (私有/团队/公开)
├── 创建初始版本 (v1.0)
└── 注册到模板库

Step 6: 验证阶段
├── 自动化测试套件执行
├── 边界情况测试 (空值/超长文本/特殊字符)
├── 性能基准测试 (< 2s 渲染)
└── 人工审核 (如启用)
```

### 3.3 翻译管道架构

**五步翻译流程详解**

```
输入: Markdown 报告 (zh-CN)
       ↓
Step 1: 结构解析 (Structure Parsing)
├── Markdown AST 解析
├── 识别文档结构层级 (H1/H2/H3...)
├── 提取纯文本片段 vs 格式标记
├── 建立位置索引映射
└── 保留代码块/表格/公式等不可翻译区域
       ↓
Step 2: 术语识别 (Terminology Recognition)
├── NER 实体识别 (专有名词/技术术语)
├── 术语库匹配查找
├── 上下文消歧 (一词多义处理)
├── 标记保护术语 (不参与通用翻译)
└── 生成术语替换表
       ↓
Step 3: 分段翻译 (Segment Translation)
├── 智能分段 (按段落/句子/语义单元)
├── LLM 翻译调用 (GPT-4/Claude/DeepL)
├── 上下文窗口管理 (保持连贯性)
├── 多模型投票 (提高准确性)
└── 翻译缓存命中检查
       ↓
Step 4: 格式重建 (Format Reconstruction)
├── 翻译后文本回填到原始结构
├── Markdown 格式恢复 (加粗/列表/代码块)
├── 表格单元格对齐调整
├── RTL/LTR 方向处理 (阿拉伯语等)
└── 特殊字符转义处理
       ↓
Step 5: 质量校验 (Quality Assurance)
├── 语法检查 (Grammar Check)
├── 术语一致性验证 (Terminology Consistency)
├── 数字/日期/单位格式校验
├── 长度比例检查 (膨胀率监控)
└── 人工抽样审核 (可选)
       ↓
输出: 翻译后的 Markdown 报告 (en-US/ja-JP/...)
```

```typescript
class PublisherFactory {
  static create(platform: PlatformType, config: PublisherConfig): IPublisher {

  }
}
```

### 5.6 数据存储方案建议

```
- 结构化查询 (按 ID/状态/日期) → PostgreSQL
- 全文检索/聚合分析 → Elasticsearch
- 版本历史/对比 → PostgreSQL (JSONB 存储)
```

### 6.3 实时协作技术方案

#### OT 算法原理

**OT (Operational Transformation) 核心思想**

OT 算法用于在分布式环境中保持多个客户端对同一文档编辑的一致性。当两个用户同时编辑同一位置时，OT 通过变换操作来协调冲突。

**转换示例图**

```
初始文档: "Hello World"

User A 在位置 5 插入 ", beautiful"
Operation A: Insert(5, ", beautiful")

User B 在位置 11 插入 "!"
Operation B: Insert(11, "!")

如果直接执行:
A 后: "Hello, beautiful World"
B 后 (基于原始): "Hello World!"  ← 丢失了 A 的更改!

使用 OT 变换:
A' = transform(B, A) = Insert(5 + len(", beautiful"), "!") = Insert(17, "!")

正确结果:
先执行 A: "Hello, beautiful World"
再执行 A': "Hello, beautiful World!"  ← 正确!
```

**协作功能清单（8项）**

| 功能          | 描述          | 优先级 | 复杂度 | MVP 包含 |
| ----------- | ----------- | --- | --- | ------ |
| **实时光标**    | 显示其他用户的编辑位置 | P0  | 低   | ✅      |
| **在线状态**    | 显示谁正在编辑     | P0  | 低   | ✅      |
| **行内注释**    | 在特定文本处添加评论  | P0  | 中   | ✅      |
| **冲突提示**    | 标记冲突区域供人工解决 | P0  | 高   | ✅      |
| **修订跟踪**    | 显示谁修改了什么    | P1  | 中   | ❌      |
| **语音/视频通话** | 集成音视频沟通     | P2  | 高   | ❌      |
| **@提及**     | 在评论中提及他人    | P1  | 低   | ✅      |
| **历史回放**    | 回放编辑过程      | P3  | 高   | ❌      |

#### 冲突解决机制

```
冲突检测 → 自动尝试合并 → 成功则静默处理

                    ↓

```

### 6.4 审批工作流引擎

#### YAML DSL 完整语法示例

```yaml
# 标准审批工作流定义
workflow:
  name: "标准报告审批流程"
  version: "1.0"
  description: "适用于大多数团队的标准报告审批流程"

states:
  - name: draft
    display_name: "草稿"
    initial: true
    transitions:
      - target: reviewing
        trigger: submit_for_review
        required_roles: [editor, owner]
        actions:
          - notify_reviewers

  - name: reviewing
    display_name: "审核中"
    transitions:
      - target: approved
        trigger: approve
        required_roles: [reviewer, admin]
        conditions:
          min_approvals: 1
        actions:
          - notify_author
          - record_approval
      - target: revision_required
        trigger: request_changes
        required_roles: [reviewer, admin]
        actions:
          - notify_author_with_comments
      - target: rejected
        trigger: reject
        required_roles: [admin]
        actions:
          - notify_author
          - archive_report

  - name: revision_required
    display_name: "需修改"
    transitions:
      - target: reviewing
        trigger: resubmit
        required_roles: [editor, owner]

  - name: approved
    display_name: "已批准"
    transitions:
      - target: published
        trigger: publish
        required_roles: [admin, owner]
        actions:
          - publish_to_targets
          - notify_stakeholders

  - name: published
    display_name: "已发布"
    final: true

  - name: rejected
    display_name: "已拒绝"
    final: true

signatures:
  approval:
    types: [digital_signature, electronic_signoff]
    required: true
  publication:
    types: [digital_signature, official_seal]
    required_for: [published]

notifications:
  channels: [email, in_app, webhook]
  templates:
    review_request: "templates/review-request.html"
    approval_notification: "templates/approved.html"
    rejection_notification: "templates/rejected.html"
```

#### 电子签名机制（4种类型）

| 签名类型                         | 安全等级 | 适用场景      | 法律效力 | 实现方式         |
| ---------------------------- | ---- | --------- | ---- | ------------ |
| **电子签章 (Click-to-Sign)**     | 低    | 内部审批、日常确认 | 有限   | 点击确认 + 时间戳   |
| **电子签名 (Typed Signature)**   | 中    | 一般业务流程    | 中等   | 输入姓名 + IP 记录 |
| **数字签名 (Digital Signature)** | 高    | 合同、合规报告   | 高    | PKI 证书 + 哈希  |
| **生物识别 (Biometric)**         | 最高   | 高敏感文档     | 最高   | 指纹/面部/声纹     |
