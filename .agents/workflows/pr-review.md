# 代码审查清单 (PR Review Checklist)

作为 AI Code Reviewer，在进行代码审查时，请严格对照以下清单：

## 1. 安全性

- [ ] 是否包含硬编码的敏感信息（密码、API Key、Token、私钥）？
- [ ] 用户输入是否经过适当的校验与清洗？
- [ ] 是否存在 SQL 注入、命令注入、路径遍历等常见攻击面？
- [ ] 敏感数据是否加密存储、加密传输？

> **示例**：
> - ❌ `API_KEY = "sk-abc123"` 写在源码中
> - ❌ `f"SELECT * FROM users WHERE id = {user_id}"` 字符串拼接 SQL
> - ✅ 从环境变量读取密钥，使用参数化查询

## 2. 健壮性

- [ ] 是否包含必要的错误处理（try/except、Result 类型）？
- [ ] 外部调用（HTTP、DB、文件 I/O）是否有超时、重试、降级策略？
- [ ] 边界条件是否覆盖（空值、零值、最大值、并发冲突）？
- [ ] 资源是否正确释放（文件句柄、数据库连接、网络套接字）？

> **示例**：
> - ❌ `data = response.json()` 不检查 status_code
> - ❌ `file.read()` 不处理 FileNotFoundError
> - ✅ 使用 context manager + retry + timeout

## 3. 规范性

- [ ] 命名是否清晰、表意，且符合项目命名规范？
- [ ] 函数长度是否合理（建议 ≤ 50 行），职责是否单一？
- [ ] 是否有不必要的注释或缺失注释（复杂逻辑）？
- [ ] 导入语句是否按标准库 → 第三方 → 项目内部排序？

> **示例**：
> - ❌ `def f(x):` 函数名无意义
> - ❌ 单个函数 200 行，混合多个职责
> - ✅ `def calculate_shipping_cost(order: Order) -> Decimal:`

## 4. 性能

- [ ] 是否存在 N+1 查询（循环内数据库调用）？
- [ ] 大数据量操作是否考虑分页、流式处理？
- [ ] 是否有不必要的重复计算、重复渲染？
- [ ] 关键路径是否有缓存策略？

> **示例**：
> - ❌ `for user in users: db.query(f"SELECT posts WHERE user_id={user.id}")`
> - ❌ 一次性加载 100 万条记录到内存
> - ✅ 使用批量查询 + 分页 + 惰性加载

## 5. 测试

- [ ] 关键逻辑是否补充了对应的单元测试？
- [ ] 测试是否覆盖正常路径、边界条件、异常路径？
- [ ] 测试是否独立（不依赖执行顺序、外部状态）？
- [ ] 新增功能是否对应新增测试用例？

> **示例**：
> - ❌ 纯 UI 变更可能不需要单元测试
> - ❌ 核心算法函数无任何测试覆盖
> - ✅ 每个 public 函数至少 1 个 happy path + 1 个 edge case 测试

## 审查结论

审查完成后，就每个维度给出结论：

| 维度 | 结论 | 说明 |
|------|------|------|
| 安全性 | ✅ 通过 / ⚠️ 建议改进 / ❌ 必须修改 | 具体问题说明 |
| 健壮性 | ✅ 通过 / ⚠️ 建议改进 / ❌ 必须修改 | 具体问题说明 |
| 规范性 | ✅ 通过 / ⚠️ 建议改进 / ❌ 必须修改 | 具体问题说明 |
| 性能 | ✅ 通过 / ⚠️ 建议改进 / ❌ 必须修改 | 具体问题说明 |
| 测试 | ✅ 通过 / ⚠️ 建议改进 / ❌ 必须修改 | 具体问题说明 |

> **使用说明**：审查结果应在 PR 评论中以结构化格式呈现，每个 ⚠️/❌ 附带对应代码位置和修改建议。
