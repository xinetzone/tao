# Spec

## Goal

- 验证一次探索闭环结束后，协议页、场景目录、导航入口、工作台与复盘之间的关键引用关系，能否用低成本、只读方式检查出来。

## Background

- 上一轮 `exploration-knowledge-loop-pilot` 已跑通“场景卡 -> spec -> plan -> 验证 -> 复盘 -> 回流动作”。
- 上一轮复盘指出，当前仍需要人工逐项核对协议页、模板、场景目录、导航入口与复盘文件之间的引用关系。
- 本轮探索优先验证“什么值得被自动化”，而不是提前构建复杂工作流。

## Scope

- 定义探索闭环引用完整性的最小必查集合。
- 使用上一轮 `exploration-knowledge-loop-pilot` 作为样本完成一次手工只读检查。
- 定义检查结果的最小状态：`PASS`、`WARN`、`MISSING`。
- 输出本轮复盘并明确是否需要进入脚本化或模板升级。

## Non-Goals

- 本轮不自动修改任何文档。
- 本轮不自动创建复盘、场景卡或工作台。
- 本轮不引入数据库、UI、服务进程或长期后台任务。
- 本轮不检查所有 Markdown 链接，只检查探索闭环所需的关键关系。
- 本轮不把检查器做成完整工作流系统。

## Core Loop

- 场景卡来源：长期场景目录中的 `探索任务知识闭环最小试点`。
- Spec 来源：本文件描述只读检查探索边界。
- Plan 来源：执行计划拆解检查规则、样本验证与复盘动作。
- 验证方式：使用上一轮试点文件进行一次手工只读检查。
- 回流方式：根据检查结果决定更新协议、模板、检查清单或后续脚本计划。

## Minimal Reference Set

- 协议页存在并能指向模板或试点工作台。
- 场景目录包含对应试点场景。
- `.trae/specs/<topic>/` 包含 `spec.md`、`tasks.md`、`checklist.md`。
- AI 文档导航能引导到探索协议或场景目录。
- 复盘文件位于 `.agents/docs/superpowers/retrospectives/`。
- 复盘文件包含 `Next Action`，且至少列出一个可执行回流动作。

## Check Status Semantics

- `PASS`：关键文件或引用存在，且满足当前最小闭环要求。
- `WARN`：文件存在但语义不够强，例如回流动作不够明确或引用只能间接追溯。
- `MISSING`：关键文件、目录、章节或引用缺失，导致闭环无法被确认。

## Deliverables

- 第二轮 `.trae/specs/exploration-reference-integrity-check/` 工作台。
- 一份执行计划。
- 一份手工只读检查记录。
- 一份复盘与至少一个回流动作。

## Validation

- 低摩擦：检查规则可以用手工阅读完成，不依赖新工具。
- 可复用：检查集合可复用于下一轮探索主题。
- 可回流：检查结论能明确指向协议、模板、清单或脚本计划。
- 可扩展：后续如脚本化，也能从当前 `PASS`、`WARN`、`MISSING` 状态语义自然演进。

## Risks

- 检查集合过大，会退化为通用 Markdown 链接检查。
- 检查集合过小，会无法发现真实断链问题。
- 如果手工检查记录不结构化，后续很难判断是否值得自动化。

## Rollback Or Adjustment

- 如果本轮检查规则过重，优先减少到只保留工作台、复盘和 Next Action 三个检查点。
- 如果本轮检查发现不了任何有效问题，优先补充 `WARN` 语义，而不是立刻扩大检查范围。

## Next Step

- 根据本 spec 创建执行计划，并用上一轮试点作为样本完成一次手工只读检查。
