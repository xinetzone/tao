# Agent 系统运行时闭环

> 来源洞察：从 Hello-Agents 教程知识萃取任务中提炼出的 AgentForge 架构参考。
>
> 相关参考：[`projects/hello-agents/index.md`](./projects/hello-agents/index.md)、[`agent-memory-dream-protocol.md`](./agent-memory-dream-protocol.md)、[`agent-collaboration-metamodel.md`](./agent-collaboration-metamodel.md)

## 定位

本文用于沉淀 AgentForge 对 Agent 系统运行时闭环的长期设计洞察。它不是某个具体框架的使用说明，也不是要求项目立即实现完整 Agent Runtime 的实施计划，而是帮助后续设计者判断：一个真正可演进的 Agent 系统，除了文档约定、上下文规则和知识归档之外，还需要逐步形成任务执行、评估反馈、记忆回流和能力演化之间的闭环。

Hello-Agents 对 AgentForge 的价值不只在于 ReAct、RAG、MCP、A2A、Agentic RL 等单点知识，而在于它展示了 Agent 系统从任务理解到多智能体应用的连续结构。这个结构提示 AgentForge：如果只停留在文档规范层，系统可以变得更易读、更可检索，但还不能自然证明 Agent 是否执行得更好、失败原因是否被捕获、经验是否能回流、下一次任务是否真的受益。

因此，AgentForge 后续的演化方向应从“文档知识库”逐步走向“可观测、可评估、可反馈、可演化的 Agent 工作系统”。

## 核心洞察

AgentForge 不应只做文档约定层，也应逐步拥有“评估—反馈—演化”的运行时闭环意识。

这句话包含三层含义：

| 层级 | 含义 | 对 AgentForge 的要求 |
|---|---|---|
| 文档约定层 | 通过规则、索引、references、sources、memories 等资产降低理解成本 | 让 Agent 知道去哪里找上下文、如何归档知识、如何遵守项目结构 |
| 运行执行层 | 在具体任务中完成推理、工具调用、文件操作、验证和交付 | 让 Agent 的行动过程可追踪，任务结果可验证，失败路径可复盘 |
| 演化反馈层 | 将执行结果、验证信号、失败模式和洞察回流到长期资产 | 让下一次 Agent 执行不是从零开始，而是继承经验并减少重复成本 |

没有文档约定层，Agent 很难稳定理解项目；没有运行执行层，文档只是静态知识；没有演化反馈层，系统每次任务都像一次孤立执行，无法累积智能。

## 运行时闭环模型

Hello-Agents 展示的 Agent 能力链路可以抽象为以下闭环：

```{mermaid}
flowchart LR
    A["任务理解"] --> B["推理范式"]
    B --> C["工具调用"]
    C --> D["记忆检索"]
    D --> E["上下文工程"]
    E --> F["协议通信"]
    F --> G["评估优化"]
    G --> H["多智能体案例"]
    H --> I["反馈回流"]
    I --> D
    I --> B
    I --> A
```

这个模型的关键不是线性执行，而是回流：

- 任务理解产生的约束应进入执行计划和验收标准。
- 推理范式的选择应受任务类型、风险、工具可用性和历史经验影响。
- 工具调用结果不应只用于当前回答，也应成为可追踪的执行证据。
- 记忆检索应服务当前任务，同时在任务结束后接收新的高价值记忆。
- 上下文工程既要压缩历史，也要为下一轮执行保留结构化状态。
- 协议通信让工具、Agent、外部服务和项目知识库可以形成稳定接口。
- 评估优化负责判断结果是否真的变好，而不是只看任务是否完成。
- 多智能体案例暴露协作、角色、状态和冲突处理问题，倒逼闭环设计成熟。

## Hello-Agents 到 AgentForge 的映射

| Hello-Agents 内容 | AgentForge 可借鉴方向 | 运行时闭环含义 |
|---|---|---|
| ReAct / Plan-and-Solve / Reflection | Agent 执行循环 | 将“思考—行动—观察—修正”从提示技巧提升为任务执行结构 |
| Memory / RAG | 长期记忆与知识回流 | 将历史经验、项目知识和任务结果从上下文窗口外部按需召回 |
| Context Engineering | 上下文路由与压缩 | 将上下文视为可治理资源，而不是无限追加的聊天历史 |
| MCP / A2A / ANP | 协作协议和外部工具接入 | 将工具、Agent、资源和网络连接协议化，减少隐式耦合 |
| Agentic RL | 自进化与反馈优化 | 将任务反馈转化为策略改进、工具选择改进或提示改进 |
| Evaluation | 验证闭环和质量度量 | 用任务集、指标、评审和回归验证证明系统能力变化 |
| 多智能体案例 | Team / Role / Agent 元模型落地 | 在复杂任务中检验角色边界、任务分发、状态共享和结果合并 |

这个映射说明，AgentForge 的文档体系已经覆盖了运行时闭环的一部分前置能力，例如规则、references、memories、retrospectives 和项目索引。但要进一步接近 Agent 系统，还需要把这些静态资产与具体执行过程、验证结果和反馈机制连接起来。

## 对 AgentForge 的设计启发

### 文档约定层只是入口

AgentForge 当前的优势在于能把外部知识、项目规则、复盘经验和长期记忆组织成结构化资产。这使 Agent 能更快进入项目上下文，也能减少重复解释。

但文档约定层本身不回答以下问题：

- 本次任务执行是否符合预期？
- 哪些工具调用有效，哪些工具调用浪费上下文？
- 哪些失败模式重复出现？
- 哪些复盘结论真的改变了下一次执行？
- 哪些记忆应继续保留，哪些应遗忘或回流为规则？

这些问题需要运行时闭环处理。

### 运行时状态需要被显式化

Agent 执行复杂任务时，状态通常散落在对话、工具结果、临时文件、todo、diff、测试输出和最终回答中。若状态不被显式化，后续复盘只能依赖回忆，长期改进会变得不稳定。

AgentForge 可以逐步将以下状态对象标准化：

| 状态对象 | 作用 | 可落地位置 |
|---|---|---|
| 任务目标 | 说明要完成什么、不要做什么 | spec、plan、todo、最终报告 |
| 执行轨迹 | 记录关键搜索、编辑、验证和决策 | retrospectives、任务摘要 |
| 验证证据 | 证明结果经过检查 | 测试输出、lint/typecheck、linkcheck、diff check |
| 失败模式 | 记录错误、触发条件和修复方式 | issue-patterns、retrospectives |
| 记忆候选 | 提炼可复用经验 | superpowers/memories |
| 回流结果 | 将稳定经验进入 references 或 rules | references、rules、workflow |

### 评估应成为闭环中心

没有评估，反馈就容易变成主观感受；没有反馈，演化就无法证明有效。

AgentForge 可以先从轻量评估开始：

- 文档任务：结构检查、链接检查、关键词覆盖、读者可发现性。
- 代码任务：lint、typecheck、测试、构建、diff 检查。
- 知识萃取任务：目录覆盖、来源追溯、核心概念、操作步骤、案例覆盖、索引入口。
- 复盘任务：是否提炼出可回流的规则、参考页、记忆或工作流。

评估结果不应只出现在最终回答中，还应成为后续复盘和记忆回流的输入。

### 记忆回流是运行时闭环的知识出口

AgentForge 已经有 `superpowers/memories/` 和 `agent-memory-dream-protocol.md`。这说明项目不仅关心“记录发生了什么”，也关心“哪些经验值得留下”。

运行时闭环中的记忆回流可以遵循：

```{mermaid}
flowchart LR
    A["任务执行"] --> B["验证结果"]
    B --> C["复盘报告"]
    C --> D["记忆候选"]
    D --> E["做梦重组"]
    E --> F["回流到 references / rules / workflow"]
    F --> G["影响下一次任务执行"]
    G --> A
```

这条链路让 AgentForge 从“文档仓库”变成“会积累经验的工作系统”。

## 可落地的最小闭环

AgentForge 不需要一开始就构建复杂的多智能体运行平台。更合理的起点是先把已有文档治理能力串成最小闭环：

```text
任务 → 执行记录 → 验证结果 → 复盘 → 记忆候选 → references/rules 回流 → 下一次任务
```

可拆解为六个动作：

| 步骤 | 最小动作 | 目标 |
|---|---|---|
| 1. 任务进入 | 明确目标、范围、验收标准 | 避免执行偏移 |
| 2. 执行记录 | 保留关键决策、搜索路径、修改范围 | 让复盘有证据 |
| 3. 验证结果 | 运行可用检查并记录异常 | 让完成状态可证明 |
| 4. 复盘提炼 | 总结有效做法、错误和改进点 | 形成经验候选 |
| 5. 记忆归档 | 将长期有效原则写入 memories | 降低未来重复成本 |
| 6. 知识回流 | 将成熟记忆迁移到 references、rules 或 workflow | 让经验参与下一次执行 |

这个最小闭环符合 AgentForge 当前文档结构，不要求引入额外运行时系统，也不会过早绑定某个 Agent 框架。

## 风险与边界

### 避免过早平台化

运行时闭环意识不等于立刻实现完整 Agent Runtime、Agent 调度平台或多智能体网络。过早平台化会带来接口、状态、持久化、评估和安全边界的复杂度，可能反而拖慢项目知识治理的演进。

当前更适合先强化“文档化闭环”和“验证闭环”，再根据高频任务逐步抽象自动化能力。

### 避免把闭环误解为自动化流水线

闭环的本质不是把所有步骤自动串起来，而是确保每一步的产物能被下一步消费：

- 执行记录能被复盘消费。
- 验证结果能被质量判断消费。
- 复盘洞察能被记忆系统消费。
- 长期记忆能被 references、rules 或 workflow 消费。
- 回流后的知识能被下一次任务消费。

只要这些消费关系稳定存在，即使过程仍由人工或 Agent 半自动执行，也已经形成闭环。

### 避免缺少遗忘机制

如果所有经验都无差别进入长期记忆，知识库会逐渐膨胀并降低检索质量。运行时闭环必须同时包含遗忘、过期条件和回流判断。

因此，memory 条目应继续保留适用范围、失效条件和回流建议。成熟经验应进入 references 或 rules；临时经验应留在 retrospectives；过期经验应被标记或移除。

## 适用场景

优先使用本文的场景：

- 设计 AgentForge 的 Agent Runtime、任务执行器、工作流或协作协议。
- 判断某个复盘洞察是否应进入长期记忆或 references。
- 为复杂任务设计验证、反馈和知识回流路径。
- 将外部 Agent 教程、框架或案例映射到 AgentForge 架构。
- 评估 AgentForge 是否只是“文档组织系统”，还是已经具备“执行后可演化”的闭环能力。

不适合使用本文的场景：

- 只需要查找某个具体命令或工具用法。
- 只需要处理一次性网页摘要。
- 尚未明确任务边界，却试图设计完整 Agent 平台。
- 需要具体实现代码而不是架构判断。
