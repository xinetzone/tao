# 2. AGENTS.md 的核心定位

## 2.1 仓库级 AI Agent 协作协议

`AGENTS.md` 的价值体现在四个层面：

1. **给 Agent 明确项目规则** — 技术栈、架构边界、代码风格、测试命令、禁止事项、安全约束。
2. **减少每次对话重复说明** — 不用每次都告诉它"不要改这个目录""测试要跑这个命令""我们用 pnpm 不用 npm"。
3. **让不同 Agent / IDE / CLI 工具共享上下文** — Claude Code、OpenAI Codex CLI、Cursor、Continue、Aider、Devin-like agents、自研 Multi-Agent 系统均可读取。
4. **降低 AI 误操作风险** — 明确哪些文件不能碰、哪些命令不能运行、哪些行为必须先征求确认、不能提交 secrets、不能自动部署生产环境。

## 2.2 操作手册，而非组织叙事

很多人写 `AGENTS.md` 会写成 onboarding 文档——公司使命、系统历史、团队文化。这些对人有帮助，但对 coding agent 的执行帮助有限。Agent 更需要的是：

- 入口在哪里
- 哪些命令可以跑
- 哪些目录负责什么
- 什么不能改
- 修改后如何验证
- 当前项目偏好的实现方式
- 遇到不确定时如何决策

`AGENTS.md` 应该偏操作手册，而不是组织叙事。

## 2.3 好的 AGENTS.md 的判断标准

1. **短** — 最好 100～200 行以内，不频繁写临时任务。
2. **稳定** — 避免"本周我们正在迁移认证"这类临时内容，应放 issue/PR/spec。
3. **明确** — 不要"使用最佳实践"，应该"当 TypeScript 文件变更后，在最终回复前运行 `pnpm typecheck`"。
4. **可执行** — Agent 能根据它直接行动。
5. **分层** — 复杂内容链接到 `.agents/`。
6. **无冲突** — 不要同时写"用 npm"和"用 pnpm"。
7. **面向风险** — 明确提交、部署、迁移、删除、密钥处理等高风险行为。
