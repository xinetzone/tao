# 22. 设计原则总结

如果用一句话概括：

> 现代 `AGENTS.md + .agents/` 的最佳设计，是把仓库变成一个"Agent 可协作"的工程环境：`AGENTS.md` 提供稳定入口和硬规则，`.agents/` 提供按角色、任务、上下文、策略拆分的模块化知识。它不是单纯提示词，而是 AI 时代的工程协作协议。

如果要总结成十条可操作的原则：

1. **`AGENTS.md` 是入口，不是垃圾桶** — 只放稳定、关键、高频规则。
2. **`.agents/` 是模块化上下文，不是提示词坟场** — 每个文件有清晰职责，避免版本后缀堆积。
3. **规则要可执行，不要只写价值观** — checklist 优于"best practices"宣言。
4. **风险边界要明确** — 命令分级（Safe / Confirmation Required / Forbidden）。
5. **命令要具体** — 给出可复制运行的精确命令。
6. **上下文要分层** — AGENTS.md（高频） → .agents/context/（低频） → docs/（人类参考）。
7. **项目规范应版本化，不应只存在 Agent memory** — 可审查、可共享、可迁移。
8. **越靠近代码目录，规则越具体** — 子目录 AGENTS.md 覆盖父目录规则。
9. **不要和工具私有规则重复太多** — AGENTS.md 做中立源头，其他工具引用或同步。
10. **先 Markdown，后 schema；先轻量，后自动化** — 不要一开始就过度设计，从最小版本起步，按需演进。
