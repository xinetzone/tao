# 反模式-检测信号对照表

> **用途**：Code Review、技术方案评审、代码巡检时快速对照自查。每种反模式包含：名称、检测信号、典型案例、推荐对策。
>
> **来源**：基于对软件工程通用反模式的提炼。
>
> **使用方式**：在审查代码或方案时，逐一扫描检测信号。命中任一条即建议启动专项治理。

---

## 一、模块职责类

### 1.1 工具模块膨胀（Utility Bloat）

> 名为 `util` / `helper` / `common` / `shared` 的模块逐渐承载业务规则，成为事实上的业务逻辑容器。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 工具模块中出现了领域模型、业务枚举、或业务规则判断逻辑 |
|  | ② 同一个函数同时做了"数据获取""条件判断""结果组织"三类功能上不相关的操作 |
|  | ③ 函数名以通用动词开头（如 `copy` / `move` / `collect` / `process`），但函数体内包含了平台检测、环境探测或版本判断逻辑 |
|  | ④ 通用工具目录的文件数和行数持续增长，但没有新增业务模块与之对应 |
|  | ⑤ 两个不同的业务方都在往同一个工具模块里加函数 |
| **典型案例** | 一个名为 `FileHelper` 的模块，最初只负责文件读写，后来逐步加入了：产物后缀匹配、目标路径推导、符号链接回退策略——这些实际上是打包业务逻辑，而非通用文件操作 |
| **推荐对策** | ① 将业务规则从工具模块中抽离为独立的解析器、适配器或策略模块 |
|  | ② 工具函数保持"接受明确的输入，返回确定的结果"，不做环境探测或规则判断 |
|  | ③ 引入数据契约（如配置结构体、产物清单数据结构）替代散落的隐式假设 |
| **严重程度** | 🟡 中等（短期不影响功能，长期导致所有变更都要改"工具"模块） |

---

### 1.2 上帝对象（God Object）

> 一个类或模块承担了过多的职责，系统中几乎所有功能都经过它。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 单个类的方法数超过 20 个，且方法之间没有明显的关联 |
|  | ② 类名中包含 `Manager` / `Handler` / `Processor` / `Service` 且无法用一句话说清它不做什么 |
|  | ③ 该类的引用出现在项目中超过 50% 的业务文件中 |
|  | ④ 修改一个不相关的功能时，这个类也被改动了 |
|  | ⑤ 该类的构造函数或初始化方法接收超过 5 个外部依赖 |
| **典型案例** | 一个 `OrderService` 既处理下单、又处理支付回调、又处理退款、又处理发票——实际上每个子域对应独立的业务上下文，却被耦合在同一个类中 |
| **推荐对策** | ① 按业务上下文拆分为多个独立模块或类 |
|  | ② 使用 facade 模式对外暴露统一入口，内部委托给各子模块 |
|  | ③ 如果暂不能拆分，至少按功能域将方法分组并加注释标明边界 |
| **严重程度** | 🔴 严重（直接导致改动风险高、测试困难、新人上手周期长） |

---

### 1.3 霰弹式修改（Shotgun Surgery）

> 改动一个概念需要同时修改多个文件，变动散弹般扩散。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 新增一个平台/环境/版本支持，需要改动超过 3 个文件 |
|  | ② 同一类常量（如版本号、文件名模式、路径前缀）在 3 个以上文件中各自定义 |
|  | ③ PR 的变更文件列表中频繁出现"改 A 就一定同时改 B、C、D"的固定组合 |
|  | ④ 一个配置变更需要同时更新代码、配置文件、部署脚本和环境变量 |
| **典型案例** | 新增一个部署目标时，需要同步修改构建脚本、打包模块、发布模块、校验模块中的文件匹配规则和路径拼接逻辑——这些分散在各处的规则本应共享同一个定义 |
| **推荐对策** | ① 将变化原因相同的逻辑收敛到同一模块，形成单一变更点 |
|  | ② 使用配置表或注册机制代替分散的硬编码 |
|  | ③ 在方案评审阶段就对"这个变更需要改哪些文件"进行影响面分析 |
| **严重程度** | 🟡 中等（单次改动成本可控，但累积后会成为重构的最大心理障碍） |

## 二、错误处理类

### 2.1 静默失败传播（Silent Failure Propagation）

> 底层模块的失败被吞掉，上层调用方在不知道失败的情况下继续执行。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 函数内部存在捕获所有异常后吞掉的模式（如 `catch` 块为空或只返回默认值） |
|  | ② 资源或文件缺失时函数不报错，而是返回空列表、空字符串或静默跳过 |
|  | ③ 调用方在执行某个操作后紧跟一条无条件的成功日志，而该操作本身没有返回状态 |
|  | ④ 流水线或编排脚本中前一阶段报错但后一阶段仍然启动并报告"完成" |
|  | ⑤ 函数返回值中没有区分"成功""部分成功但不完整""完全失败"的能力 |
| **典型案例** | 一个文件收集函数在找不到目标文件时不抛出异常也不返回错误信息，而是静默返回空列表。上游调用方在调用后直接打印"收集完成"，整个流水线继续走到发布阶段，最终产出一个缺件的包 |
| **推荐对策** | ① 将"可能部分成功"的操作的返回值改为结构化结果对象（包含成功列表、失败列表、警告列表） |
|  | ② 区分两种失败级别：可降级继续的警告（warn）和必须终止的错误（error） |
|  | ③ 调用方基于返回结果显式决策，而非依赖"没有异常就等于成功"的假设 |
|  | ④ 对关键步骤增加"操作后校验"（如收集完成后立即检查产物数量） |
| **严重程度** | 🔴 严重（直接导致残缺产物进入下游环节，问题定位需回溯整个流水线） |

---

### 2.2 异常吞噬（Exception Swallowing）

> 捕获了过于宽泛的异常类型，且未做任何有意义的处理或记录。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 通用异常捕获（`catch (Exception)`、`except Exception`、`rescue`）出现在非最外层代码中 |
|  | ② 异常捕获块中只有"吃掉"操作——不重新抛出、不记录日志、只返回默认值 |
|  | ③ 日志中缺少对应的异常记录，但功能行为表现异常或不完整 |
|  | ④ 同样的异常捕获模式在不同语言/模块中重复出现，但没有统一的错误处理策略 |
| **典型案例** | `try { file.read(); } catch { return ""; }` —— 权限不足、磁盘满、文件被删除三种机制完全不同的问题被等同处理，排查时无法判断到底发生了什么 |
| **推荐对策** | ① 捕获最小范围的具体异常类型（如 `FileNotFoundError` 而非通用 `Exception`） |
|  | ② 至少记录异常上下文（异常类型、消息、堆栈）后再决定是否降级 |
|  | ③ 对于不能安全降级的异常，应向上传播而非吞掉 |
| **严重程度** | 🔴 严重（直接掩盖问题根因，大幅延长故障发现和修复时间） |

---

### 2.3 失败语义缺失（Missing Failure Semantics）

> 函数只能通过"抛异常/不抛异常"表达失败，无法区分"完全成功""部分成功""降级成功""完全失败"。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 函数返回 `boolean` 或 `void/null`，但调用方实际需要知道"哪些项成功了、哪些失败了、为什么失败" |
|  | ② 调用方用异常捕获包裹函数调用，但捕获块中的处理与正常返回后的处理逻辑完全一致 |
|  | ③ 批量操作的日志中，整体显示"成功"但实际某些子项被静默跳过了 |
|  | ④ 函数文档中写"调用方需自行检查结果"，但可检查的信息不足以判断原因 |
| **典型案例** | 一个批量部署函数返回 `true/false`。某次部署中 10 个目标成功了 9 个，返回 `false`——调用方只知道"失败了"，不知道是哪个目标失败、失败原因是什么、已成功的 9 个是否需要回滚 |
| **推荐对策** | ① 设计包含详细信息的返回类型：`succeeded[]`、`failed[]`（含原因）、`warnings[]` |
|  | ② 对自然批量的操作使用"全量执行 + 汇总报告"模式，而非"遇错即停" |
|  | ③ 在方案评审阶段就明确函数契约：调用方拿到返回值后能做什么决策 |
| **严重程度** | 🟡 中等（影响故障定位效率和回滚决策速度，但通常不直接导致线上事故） |

## 三、规则与配置类

### 3.1 规则重复定义（Rule Duplication）

> 同一个业务规则在系统的不同环节有各自独立的解释逻辑，且相互不知晓。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 描述同一概念的常量、正则表达式或匹配模式出现在两个以上文件中 |
|  | ② 修改一个模块的匹配规则后，另一个模块不报错但行为出现偏差 |
|  | ③ 两处代码处理"同一件事"但使用了不同的判断逻辑（如一处用宽松匹配，一处用精确匹配） |
|  | ④ 文档或规范中写了一套约定，但代码中有另一套独立实现 |
|  | ⑤ 代码审查时频繁出现"这个地方的值和 X 模块保持一致吗"类的问题 |
| **典型案例** | 构建阶段用通配符匹配产物文件名，打包阶段用固定字符串匹配同一批产物。版本升级后只更新了构建阶段的匹配规则，打包阶段因仍使用旧规则导致找不到文件——两处规则本应是同一个定义的两个消费方，却各自维护了一份副本 |
| **推荐对策** | ① 将规则定义提取为单一数据源，所有消费方通过引用获取 |
|  | ② 引入解析器/适配器模式：业务方只消费解析结果，解析器统一维护规则 |
|  | ③ 代码审查时对"看起来熟悉的常量"保持警觉，用搜索确认是否已有定义 |
| **严重程度** | 🟡 中等（当前版本正常工作，但环境或版本变更时成为隐蔽陷阱） |

---

### 3.2 环境硬编码（Environment Hardcoding）

> 环境、平台、版本相关的特定值直接内嵌在业务逻辑中。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 代码中出现操作系统名称、架构标识、具体版本号等系统特定值 |
|  | ② 平台判断逻辑（`if os == 'linux'`、`#ifdef _WIN32`）散落在多个业务文件中 |
|  | ③ 文件扩展名、路径分隔符以字符串字面量形式出现 |
|  | ④ 配置条件中有"仅适用于 XX 环境"的注释，但代码无条件执行对应逻辑 |
| **典型案例** | 产物文件的匹配规则中直接嵌入了某操作系统的共享库扩展名和特定版本后缀——任何平台或版本的变更都需要修改业务逻辑代码 |
| **推荐对策** | ① 使用标准库或框架提供的平台抽象 API，而非自行拼接环境标识符 |
|  | ② 平台差异通过策略模式或依赖注入处理，而非散落的条件分支 |
|  | ③ 将环境相关的配置值集中管理，支持通过配置文件或环境变量覆盖 |
| **严重程度** | 🟡 中等（所有跨平台迁移、版本升级的第一道障碍） |

---

### 3.3 配置来源混乱（Configuration Source Chaos）

> 同一配置项在代码默认值、配置文件、环境变量、命令行参数中各自定义，优先级关系不明确。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 同一个配置值可能在多个来源中出现（代码硬编码、JSON/YAML 配置文件、环境变量、CLI 参数） |
|  | ② 修改行为后不确定"最终生效的到底是哪个来源的值" |
|  | ③ 配置变更需要在多个位置同步更新 |
|  | ④ 排查问题时，需要逐个检查各个配置来源才能还原实际运行参数 |
| **典型案例** | 输出路径在入口脚本中有默认值、在全局配置文件中可覆盖、在命令行参数中可再次覆盖、在某个子模块中又有独立的路径常量——四个来源之间的优先级从未被文档化 |
| **推荐对策** | ① 明确配置优先级链并文档化（如：CLI 参数 > 环境变量 > 配置文件 > 默认值） |
|  | ② 应用启动时打印"实际生效的完整配置"，便于排查 |
|  | ③ 所有配置项在一个集中的初始化阶段完成合并，业务代码只读取最终结果 |
| **严重程度** | 🟢 一般（通常不会直接导致事故，但排查和调试成本显著增加） |

## 四、依赖与耦合类

### 4.1 循环依赖（Circular Dependency）

> 模块 A 依赖模块 B，模块 B 又依赖模块 A，形成引用闭环。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 引用关系形成闭环（可用静态分析工具检测） |
|  | ② 两个模块的单元测试必须同时运行才能通过，无法独立测试 |
|  | ③ 单独加载某个模块时报错，但启动完整应用时正常 |
|  | ④ 重构时发现"动 A 必动 B，动 B 必动 A"的连带效应 |
| **推荐对策** | ① 提取共享接口或抽象到独立的公共模块 |
|  | ② 使用依赖反转：高层模块定义接口，低层模块实现接口 |
|  | ③ 引入事件机制或消息总线解耦直接引用 |
| **严重程度** | 🔴 严重（阻碍模块独立测试、复用和部署） |

---

### 4.2 接口隐式化（Implicit Interface）

> 模块间的约定完全依赖文档或口头沟通，代码层面无强制约束。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 注释中写"调用方应确保参数有效"，但函数内部不做校验 |
|  | ② 同样的目录结构、数据格式在 3 个以上模块中分别以字符串或硬编码形式出现 |
|  | ③ 文档中声明了一份接口契约（如产物清单、数据格式），但代码中没有任何结构来校验或执行这份契约 |
|  | ④ 模块 A 的作者修改了输出格式，模块 B 的调用方在集成测试阶段才发现不兼容 |
| **典型案例** | 一份打包规范文档中记录了产物目录的结构约定，但负责收集产物的模块并没有读取或参照这份约定——它有自己的硬编码规则。当规范更新后，收集逻辑不会自动感知到变化 |
| **推荐对策** | ① 将接口约定转化为类型定义或数据结构，使编译器/解释器能在构建阶段发现不兼容 |
|  | ② 在模块边界增加校验逻辑（schema 校验、断言、前置条件检查） |
|  | ③ 接口变更时通过类型系统的编译错误而非运行时异常来提示所有调用方 |
| **严重程度** | 🟡 中等（单次可依赖沟通协调，但团队规模扩大或人员变动后风险骤增） |

---

### 4.3 传递依赖泄露（Transitive Dependency Leak）

> 模块 A 通过模块 B 间接依赖模块 C，但模块 A 的代码中直接使用了模块 C 的接口。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 模块的引用的引用被上层模块直接使用 |
|  | ② 计划替换或移除模块 B 时，发现上层模块也依赖了模块 B 引入的组件 |
|  | ③ 包管理文件中出现了并非直接声明的依赖被代码引用 |
| **推荐对策** | ① 在模块 B 中封装对模块 C 的使用，对外暴露自己的接口而非透传依赖 |
|  | ② 使用依赖分析工具定期检查未声明的直接引用 |
| **严重程度** | 🟡 中等（升级或替换依赖时出现意外断裂） |

---

## 五、可观测性类

### 5.1 日志信号缺失（Missing Log Signals）

> 关键步骤没有日志，或日志内容无法用于排查问题。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 核心流程的入口和出口没有日志记录 |
|  | ② 异常捕获处只做了简单的信息输出而非结构化日志 |
|  | ③ 日志中只有时间戳，没有请求标识、会话 ID 或上下文信息 |
|  | ④ "成功"日志出现在操作执行之前，而非执行并验证通过之后 |
|  | ⑤ 批量操作只有一个"开始"和"结束"日志，中间每个子项的状态不可见 |
| **典型案例** | 一个批量处理函数在开始前打印"开始处理"，执行完毕后打印"处理完成"——但如果处理了 100 条数据，有 3 条失败，日志中看不出是哪 3 条以及失败原因 |
| **推荐对策** | ① 在关键步骤前后记录结构化日志（步骤名、输入摘要、输出摘要、耗时） |
|  | ② 成功确认日志放在操作验证通过后，而非操作调用前 |
|  | ③ 批量操作至少输出"总数/成功/失败/跳过"的汇总信息 |
| **严重程度** | 🟢 一般（降低排查效率，但通常不会引入新的功能性问题） |

---

### 5.2 指标缺失（Missing Metrics）

> 关键业务或技术指标未被采集，无法判断系统运行是否健康。

| 维度 | 内容 |
|------|------|
| **检测信号** | ① 无法回答"这个操作的成功率是多少"或"平均耗时是多少" |
|  | ② 问题总是由用户或下游系统报告，而非监控发现 |
|  | ③ 流水线失败后只能靠人工回溯日志，没有聚合视图 |
| **推荐对策** | ① 对关键操作埋点：调用次数、成功率、耗时分布 |
|  | ② 构建或打包流程中增加产物数量、文件大小等可校验的指标 |
| **严重程度** | 🟢 一般（影响运维和优化效率） |

## 附录

### 严重程度说明

| 级别 | 标记 | 含义 |
|------|------|------|
| 严重 | 🔴 | 直接导致线上事故、数据不一致、或安全漏洞 |
| 中等 | 🟡 | 正常运行时无影响，但变更成本高或隐藏风险 |
| 一般 | 🟢 | 影响开发效率和排查体验，但不直接影响功能正确性 |

### 使用建议

| 场景 | 使用方式 |
|------|---------|
| **Code Review 前** | 快速扫描"检测信号"，优先关注 🔴 级别的信号 |
| **重构规划时** | 逐条对照现有代码，识别技术债集中区域，确定治理优先级 |
| **新人入职培训** | 作为代码规范和设计原则的配套案例材料 |
| **定期代码巡检** | 每季度扫描一次，跟踪反模式的数量和分布变化趋势 |
