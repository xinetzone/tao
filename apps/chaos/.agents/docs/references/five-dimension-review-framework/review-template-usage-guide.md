# 五维度评审模板使用说明书

> **适用对象**：需要使用[tech-proposal-review-template.md]进行技术方案评审的工程师、Tech Lead 和架构师。
>
> **前置阅读**：建议先浏览模板本身，了解整体结构后再阅读本说明。

---

## 一、何时使用本模板

### 适用场景

| 场景 | 示例 | 推荐使用方式 |
|------|------|-------------|
| 评审重构方案文档 | XMNN 打包耦合重构、支付模块解耦 | 完整五个维度 |
| 评审技术选型建议书 | 消息队列选型（Kafka vs RocketMQ）、序列化方案选型 | 维度一调整为"问题定义" |
| 评审架构设计文档 | 微服务拆分方案、读写分离设计 | 重点用维度二、三、四 |
| 团队技术方案评审会 | 周度/双周度的集体评审 | 评审者各自填写后汇总 |

### 不适用场景

| 场景 | 为什么不适合 | 替代建议 |
|------|-------------|---------|
| 代码改动极其微小（几行配置、一个简单 bugfix） | 评审成本远超改动风险 | 直接用 Code Review |
| 纯前端 UI 调整（不涉及架构） | 维度一、三、四相关性弱 | 设计评审 checklist |
| 故障复盘 | 需要根因分析和预防措施的专用框架 | 5 Why、鱼骨图 |

---

## 二、各维度填写指南

### 维度一：问题诊断评估

**目的**：判断文档作者是否真正理解了要解决的问题。

**填写步骤**：

1. 阅读文档中的"问题"章节
2. 对照 1.1 表格逐条打标记（✅ / ⚠️ / ❌）
3. 对照 1.2 表格判断问题深度
4. 在 1.3 补充区写下你认为遗漏的分析角度

**关键判断点**：

| 如果你发现 | 说明 | 标记建议 |
|-----------|------|---------|
| 文档只说"XX 模块耦合严重"，没说具体哪里耦合、怎么耦合的 | 问题描述模糊 | ⚠️，要求补充代码引用 |
| 文档把环境配置错误说成是代码 bug | 根本原因判断错误 | ❌，直接退回 |
| 文档承认"当前有临时方案，但无法扩展" | 区分了直接原因和根本原因 | ✅ |

**常见误区**：

- ❌ 不要在补充区写"方案应该怎么做"——那是维度二的事
- ❌ 不要因为"问题写得不够严重"而给低分——评估的是诊断准确性，不是问题严重程度

---

### 维度二：方案设计评估

**目的**：判断方案在技术上是否可行、架构上是否合理。

**填写步骤**：

1. 阅读文档中的"方案"或"设计"章节
2. 在 2.1 中评估技术选型的合理性
3. 在 2.2 中评估架构设计质量
4. 在 2.3 中识别文档未讨论的潜在风险
5. 在 2.4 补充区写下你认为可以优化的设计决策

**2.3 风险识别表的使用方法**：

该表第一列需要评审者**主动识别**风险并填写，而非从文档中查找。这是模板中最需要评审者经验和判断力的部分。

示例填写：

| 风险点 | 可能性 | 影响 | 文档是否讨论 | 评审者建议 |
|--------|--------|------|-------------|-----------|
| 新引入的 ArtifactSpec 与现有发布脚本不兼容 | 中 | 高 | ❌ | 建议在实施前先验证与 release 脚本的兼容性 |
| EXTENSION_SUFFIXES 在某些环境返回空列表 | 低 | 高 | ❌ | 建议增加显式错误处理和默认后缀回退 |

**常见误区**：

- ❌ 不要在 2.1 中把"是否引入了新模式"直接标记为 ❌——新模式本身不是问题，不合适的新模式才是问题
- ❌ 不要跳过 2.3——风险识别是评审者最有价值的贡献

---

### 维度三：回归完备性评估

**目的**：判断测试和验证策略是否足以保障改动安全上线。

**填写步骤**：

1. 阅读文档中的"测试""回归""验证"章节（如果有）
2. 在 3.1 中检查主链路覆盖
3. 对照 3.2 中的 6 类预设边界场景，逐条判断
4. 在 3.3 中补充本方案特有的边界场景

**3.2 边界场景逐条使用说明**：

| 边界场景 | 何时标记"需要补充" | 本场景不适用的情况 |
|----------|-------------------|-------------------|
| 输入为空/缺失/格式异常 | 方案涉及文件读取、API 调用、数据解析 | 纯内部逻辑调整，无外部输入 |
| 并发/重入场景 | 函数可能被多次调用、多个进程共享输出目录 | 单线程、无共享状态的脚本 |
| 权限不足/资源耗尽 | 涉及文件写入、网络请求、内存分配 | 纯内存计算、无 IO 操作 |
| 旧数据/残留文件存在 | 输出目录曾经运行过旧版本 | 每次运行前都完整清理环境的场景 |
| 平台差异 | 涉及编译产物、系统路径、扩展名 | 纯 Python 标准库、无平台依赖 |
| 降级/回退路径 | 方案是"原子替换"式改造 | 可逆改造（新旧方案可并行） |

**常见误区**：

- ❌ 不要因为"这 6 类场景很难全部覆盖"就跳过——标出文档没覆盖的，就是评审的价值
- ❌ 不要自己替文档作者补充测试方案——指出遗漏即可，方案由作者补充

---

### 维度四：收益可信度评估

**目的**：判断文档声称的改造成果是否真实可信、是否值得投入。

**填写步骤**：

1. 从文档中逐条提取收益声明，填入 4.1 表格
2. 判断每条声明是否有量化支撑（数字、基线、对比）
3. 判断每条声明事后是否可以验证
4. 在 4.2 中评估整体量化质量

**4.1 评估标准**：

| 收益声明示例 | 量化锚点 | 可验证 | 评估 |
|-------------|---------|--------|------|
| "平台差异收敛到 1 处规则定义" | ✅（"4 处→1 处"） | ✅（数代码就行） | 可信 |
| "跨平台扩展更便捷" | ❌（什么叫便捷？） | ❌（怎么测便捷？） | 需补充 |
| "代码可维护性显著提升" | ❌（形容词堆砌） | ❌ | 存疑 |

**常见误区**：

- ❌ 不要把"收益看起来很大"等同于"收益可信"——可信度取决于验证方式，不取决于数字大小
- ❌ 不要因为文档没有量化数据就直接给 ❌——先判断这个收益本身是否难以量化（如"架构清晰度提升"本身不易量化，可以要求在后续回顾中收集主观反馈）

---

### 维度五：整体评价与行动建议

**目的**：综合前四个维度，给出明确的推进结论和分阶段行动建议。

**填写步骤**：

1. 在 5.1 中回顾前四个维度的发现，为五个评价方面打分
2. 在 5.2 中列举文档的突出优势（至少 1 条）
3. 在 5.3 中按四个阶段分别给出行动建议
4. 在 5.4 中勾选最终评审结论

**5.3 四阶段建议的填写粒度**：

| 阶段 | 填写内容 | 示例 |
|------|---------|------|
| 实施前（先决条件） | 推进前必须完成的事项 | "与 compile_tvm_vta.py 维护者对齐产物检测规则" |
| 实施中（过程关注） | 实施时需要特别留意的点 | "关注 EXTENSION_SUFFIXES 不同平台的表现" |
| 实施后（验证与回顾） | 验证改动效果的方式 | "在 CI 环境中注入缺失产物，确认能正确报错" |
| 远期规划（持续改进） | 本次不必须但建议未来考虑的 | "新平台适配完成后，回顾本次架构决策的有效性" |

**5.4 评审结论的使用**：

| 结论 | 使用条件 | 对作者意味着什么 |
|------|---------|----------------|
| ✅ 通过 | 无阻塞性问题，补充项可以后续完善 | 方案可以实施 |
| ⚠️ 有条件通过 | 有若干需要修改的点，但不影响整体方向 | 修改指定内容后即可推进，无需重新评审 |
| ❌ 不通过 | 存在根本性问题（诊断错误、方案不可行等） | 需要重新设计方案 |

**常见误区**：

- ❌ 不要把"有条件通过"当成"不通过"的委婉说法——两者的区别是有没有根本性的方向问题
- ❌ 不要在建议中写"建议多写点注释"——建议要针对方案的实质性内容

## 三、评审流程建议

### 个人评审流程（推荐）

```
通读文档（15 分钟）
    ↓
逐维度填写模板（20-30 分钟）
    ↓
回头检查：所有 ⚠️ 和 ❌ 是否都有明确的改进方向？
    ↓
给出最终结论
```

### 团队评审流程（推荐）

```
会前：评审者各自填写模板（异步，约 30 分钟）
    ↓
会中：主持人汇总 ⚠️ 和 ❌ 进行集中讨论（约 20 分钟）
    ↓
会中：逐一讨论补充项中的关键发现（约 10 分钟）
    ↓
会后：主持人汇总最终结论和行动项，发给文档作者
```

### 产出物归档建议

完成评审后，将模板文件与以下信息一并归档：

- 被评审的文档名称和链接
- 评审者和评审日期
- 评审结论和关键行动项
- 如果方案通过，记录实施后的验证结果

## 四、与其他流程的衔接

### 与 Code Review 的关系

| | 技术方案评审 | Code Review |
|---|------------|------------|
| **评审对象** | 设计/方案文档 | 代码实现 |
| **发生时机** | 编码前 | 编码后 |
| **关注点** | 架构、选型、策略 | 代码质量、风格、正确性 |
| **本模板适用** | ✅ | ❌ |

Code Review 阶段应检查：代码实现是否与通过评审的方案一致。如果在 Code Review 中发现了方案未讨论的设计问题，应考虑回退到方案评审阶段。

### 与上线审批的关系

评审结论可作为上线审批的输入，但不能替代上线审批。模板评估的是"方案是否合理"，上线审批还需要考虑"时机是否合适""回滚预案是否就绪"等运维层面的问题。

## 五、模板定制指南

### 根据文档类型裁剪

| 文档类型 | 建议操作 |
|----------|---------|
| 重构方案 | 五维度全量使用 |
| 技术选型 | 维度一的"演变路径""模式识别"标为不适用；维度四重点使用 |
| 新功能设计 | 维度一的"演变路径"标为不适用；补充项可以新增与产品需求对齐相关的检查 |
| 性能优化方案 | 在维度四的收益声明中强制要求性能对比数据 |

### 新增自定义确认项

如果在多次评审中反复发现同类遗漏，可以将其加入对应维度的确认项表格或补充区。建议同时记录该确认项的来源说明，使后续使用者理解其背景。

### 调整评审粒度

| 调整方向 | 方法 |
|---------|------|
| 更轻量（5-10 分钟完成） | 只勾选 ✅/⚠️/❌，不填依据列；只填补充项中有实质性发现的内容 |
| 更深入（适合关键技术决策） | 每个 ⚠️ 和 ❌ 都写详细说明；在补充项中展开技术建议 |
