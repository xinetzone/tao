# 复盘报告命名与格式规范

本文档定义项目中所有复盘报告（Retrospectives / Postmortems / Task Execution Summaries）的统一命名规则、章节结构与质量标准。

## 1. 归档位置

所有复盘报告**必须**存放在：

```
.agents/docs/superpowers/retrospectives/
```

## 2. 命名规范

### 2.1 标准格式

```
{type-prefix}-{kebab-description}-{YYYYMMDD}.md
```

| 组成部分 | 说明 | 示例 |
|---------|------|------|
| `type-prefix` | 复盘类型标识 | `task-summary` / `retrospective` |
| `kebab-description` | 小写短横线连接的描述性名称 | `changelog-modularization` |
| `YYYYMMDD` | 报告生成日期 | `20260521` |

### 2.2 类型前缀一览

| 前缀 | 适用场景 | 示例 |
|------|---------|------|
| `task-summary-` | 单次 Spec 执行总结 | `task-summary-mise-dev-environment-20260522.md` |
| `retrospective-` | 项目级 / 多任务综合复盘 | `agentforge-project-retrospective-20260523.md` |
| `insights-` | 从任务中提炼的设计反思 / 原则洞察 | `insights-containerrun-refactor-20260610.md` |
| （无前缀） | 专项审查 / 审计 / 适配报告 | `python315-adaptation-20260521.md` |

### 2.3 命名反例

| 反例 | 问题 | 正确写法 |
|------|------|---------|
| `总结.md` | 无日期、无前缀 | `task-summary-xxx-20260523.md` |
| `复盘报告-2026-05-23.md` | 中文命名、日期格式不规范 | `retrospective-xxx-20260523.md` |
| `report_v2_final.md` | 无结构化标识 | 使用标准格式 |

## 3. 章节结构

### 3.1 必选章节（所有复盘报告）

所有复盘报告至少需包含以下核心章节：

1. **基本信息**：复盘对象、时间范围、参与方
2. **执行概览**：核心数据摘要（任务数、通过率、关键指标）
3. **问题与风险**：分级（P0-P4）+ 根因分析
4. **经验教训**：成功要素 + 失败教训 + 可复用方法论
5. **后续行动项**：编号、优先级、责任对象、验收方式

### 3.2 扩展章节（推荐）

对于项目级综合复盘，建议额外包含：

- 目标达成度评估
- 过程时间线与里程碑
- 成果质量审计
- 协作效能分析

### 3.3 规则候选标记（强烈推荐）

每次复盘**应主动反思**：本次任务中是否出现了可能上升为宇宙层规则的经验。该字段与 [`../rules/rule-evolution.md`](../rules/rule-evolution.md) 中的五维准入标准（频率 / 普适性 / 可执行性 / 无害性 / 可验证性）对齐，结构如下：

```markdown
## 规则候选标记

| 候选经验 | 触发次数 | 准入维度评估 | 建议动作 |
|---------|---------|------------|---------|
| <一句话描述> | <首次 / 第 N 次> | 频率□ 普适□ 可执行□ 无害□ 可验证□ | <记录 / 标记候选 / 提炼草案> |
```

填写要求：

- 若**无候选**经验，须显式声明 "无候选" 并简述原因，不得直接省略本章节。
- "建议动作"对应规则演化操作流程：**记录**（停留在 `.temp/` / retrospective）→ **标记候选**（移入 `docs/topics/`）→ **提炼草案**（PR 中起草 `.agents/rules/*.md`）。
- 五维评估缺一即不应进入 `.agents/rules/`，应停留在经验层。

### 3.4 数据引用规范

- 所有数据陈述必须标注来源（文件路径、对话片段、命令输出）
- 推测性判断需标注置信度（【高置信度】/【低置信度】）
- 禁止无依据的定性结论

## 4. 行动项格式

每个行动项必须包含以下字段：

```markdown
| # | 行动项 | 优先级 | 责任对象 | 触发条件 | 验收方式 |
|---|--------|--------|---------|---------|---------|
| A1 | 具体可执行的操作 | P0-P4 | 人类/AI/协作 | 何时执行 | 如何验证 |
```

## 5. 优先级定义

| 级别 | 含义 | 示例 |
|------|------|------|
| P0 | 阻塞性，必须立即修复 | 安全漏洞、数据丢失风险 |
| P1 | 高优先级，本次迭代内完成 | 配置不一致、功能缺陷 |
| P2 | 中等优先级，下个迭代完成 | 文档缺失、代码异味 |
| P3 | 低优先级，技术债务 | 优化建议、重构方向 |
| P4 | 建议性，可延后 | 探索性改进、nice-to-have |
