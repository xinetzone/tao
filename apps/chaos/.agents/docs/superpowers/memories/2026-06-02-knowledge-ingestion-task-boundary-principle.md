# 知识入库任务的边界控制原则

## Type

原则 / 经验 / 约束

## Scope

- 适用项目：AgentForge 及任何使用 AI 智能体进行外部资料学习、教程萃取、知识归档和文档维护的项目
- 适用场景：外部教程页面萃取、官方文档入库、知识资产归档、文档校验修复、任务复盘后提炼长期洞察
- 不适用场景：一次性网页摘要、用户只要求即时问答、无需归档的临时阅读、单文件简单编辑

## Content

- 记忆内容：
  - 外部知识入库任务的核心不是“读取并总结”，而是完成一次知识资产转化：外部材料必须经过项目语义重组、归档位置选择、索引接入和验证闭环，才能成为可被后续人类与智能体复用的资产。
  - 这类任务天然会跨越多个边界：临时抓取 vs 长期归档、源内容 vs 项目语义、任务成果 vs 额外经验、验证修复 vs 无关改动。因此执行时必须显式管理边界，而不是只追求把内容写完。
  - 复盘记录“本次发生了什么”，长期记忆记录“未来可复用的原则是什么”。当用户要求“洞察”时，应优先抽取可迁移原则并归档到 `superpowers/memories/`；当用户要求“复盘”时，才归档到 `superpowers/retrospectives/`。
  - 验证命令暴露的既有问题可顺手修复，但必须与主任务拆分为独立原子提交。知识萃取、内链修复、格式化、经验沉淀应分别提交，避免审查和回滚时边界混乱。
  - 自动格式化工具可能扩大变更范围。运行会自动修复的 lint 前，应预期它可能产生额外改动；如果产生，应单独记录、单独提交或单独说明。
  - 长期记忆文件不应由实施过程随意生成。除非用户明确要求归档洞察、复盘或长期经验，否则额外思考材料应优先放入 `.temp/`，或在最终回复中作为建议提出。
  - 单纯阅读和整理 AI/LLM 相关文档，不等于需要配置 LLM API。只有当任务要实现模型调用、接入供应商、运行依赖 API key 的功能时，LLM 配置才应成为前置条件。

- 为什么值得记忆：
  - 本次 DeepAgents overview 教程萃取任务证明，一个看似简单的“学习网页并归档”需求，实际会牵动 spec-driven 流程、网页抓取、知识重组、文档治理、链接校验、lint 自动修复、原子提交和复盘沉淀。
  - 如果没有显式边界控制，任务容易扩散为混杂提交、未经请求的长期产物或被不必要的工具配置阻塞。
  - 该洞察可复用于未来所有外部知识入库任务，帮助智能体在“做完整”和“不越界”之间保持平衡。

- 它未来能降低什么成本：
  - 降低范围判断成本：快速区分临时产物、长期资产、复盘和记忆。
  - 降低提交审查成本：按知识萃取、校验修复、格式化和洞察沉淀拆分原子提交。
  - 降低维护成本：避免 `.temp/`、占位链接、伪链接和额外经验文档污染长期知识库。
  - 降低工具误用成本：避免把普通文档学习任务误判为需要 LLM API 配置的实现任务。
  - 降低后续协作成本：让复盘和洞察分别承担不同职责，减少信息重复和归档混乱。
