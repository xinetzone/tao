### 5.2 问题 I1：C1-1 合并漏检

**问题描述**：在比较 python315-adaptation 两个文件时，仅用 `grep "核心语言特性"` 做关键词匹配，发现 task-summary 文件有两处命中即判定"已覆盖"。实际上短文件的 §4「关键发现」是完整的结构化分析（PEP 价值表 + CVE 清单），而 task-summary 文件中只是背景描述中的分类枚举。

**根因**：AI 进入"删除冗余文件"的问题框架后，产生确认偏见——倾向寻找支持删除的证据而非客观审计。这是 §2.1（命名审计）同一模式的再现。

**解决方案**：新增 `documentation.md` §2.2「文件去重」规则，要求删除前执行章节级结构对比 + 两个自检锚点问题。

**经验教训**：
- ✅ 关键词提及（如背景描述中的分类枚举）≠ 章节级覆盖（如 PEP 价值评估表 + CVE 清单）
- ✅ 规则需要"自检锚点"——将认知重操作（"逐节对比全部章节"）转化为认知轻操作（"自问两个问题"）
- ✅ 同一模式的重复违规说明单靠规则文本不足以形成拦截——需要锚点 + 违规记录的双层机制

### 5.3 问题 I3：审计工具系统性误报

**现象**：C 类（内容重复）、C2 类（内容不完整）、F 类（流转断裂）共 12 项疑似债务，经深度审计后全部确认为误报。

**根因分析**：

| 误报类型 | 工具假设 | 现实反例 |
|---------|---------|---------|
| 同名主题=重复 | 编辑距离近 → 内容重复 | 同主题可有不同维度（过程/产出）、不同会话（主/延续）、不同类型（月度/子任务/全量） |
| 行数<中位数=未完成 | 短文件=草稿 | 22-54 行的 plan/spec 可能是已实施的轻量设计 |
| 缺 Spec=断裂 | Plan 必须有 Spec | 直接实施 + 复盘记录 + 规则文件均可替代 |

**启示**：自动化审计工具的三种假设在 retrospective 类知识库中均不成立。retrospective 的特点是多维度、多会话、轻量设计——这些恰是机械化检测的盲区。

## 第九章：经验总结与方法论

### 9.1 核心方法论

#### 方法论 1：审计→治理→规则演化 三阶段闭环

**适用场景**：知识库/文档目录的债务审计与清理。

**操作步骤**：
1. **审计**：建立分类体系（四类九型）→ 机械化检测 + 人工抽样 → 输出疑似债务清单
2. **治理**：按优先级逐项深度审计（内容阅读 + 证据排查）→ 区分真实债务与误报 → 执行处置
3. **规则演化**：识别治理过程中暴露的认知盲区或工具缺陷 → 新增/强化预防规则 → 配自检锚点

**关键成功要素**：
- 分类体系必须先于检测——没有分类就无法识别系统性偏见
- 优先级驱动——P0 快速清零建立 momentum，P2-P3 批量处理节省轮次
- 自检锚点是规则生效的关键——将认知重操作转化为认知轻操作

#### 方法论 2：文件去重三段验证

**适用场景**：发现疑似重复文件需合并删除时。

**操作步骤**：
1. **章节级结构对比**：列出两文件全部标题，逐节确认对应关系
2. **三级分类**：✅ 完全覆盖 / ⚠️ 部分覆盖（需合并） / ❌ 缺失（禁止删除）
3. **自检锚点**：两个必问问题——① 同等深度？② 独有结构化数据？

**关键规则**：关键词提及 ≠ 章节级覆盖。`grep` 命中背景描述 ≠ 目标文件整章已覆盖。

### 9.2 最佳实践

#### ✅ 做得好的

1. **优先级驱动治理**：P0→P1→P2→P3→S1 分轮推进，每轮聚焦 2-4 项，避免上下文过载。
2. **子智能体并行排查**：P1 三项的证据排查同时派发 search agent，大幅节省时间。
3. **规则自愈**：发现问题后立即写入规则 + 配自检锚点，而非仅记录教训。
4. **审计报告持续更新**：每轮治理完成后同步更新报告状态，保持单一事实来源。

#### ⚠️ 需要改进的

1. **确认偏见**：首次 C1-1 比较时出现"关键词匹配=已覆盖"的误判。新 §2.2 规则可预防但需在实践中验证。
2. **删除工具依赖**：需预判 DeleteFile 在某些路径不可用，提前准备 Python 替代方案。

### 9.3 审计工具反思

本次治理的核心教训不是"某些文件需要删除"，而是**自动化审计工具的假设在知识密集型目录中系统性地失效**：

| 工具假设 | 为何失效 | 正确态度 |
|---------|---------|---------|
| 编辑距离小 = 重复 | retrospective 的多维度/多会话/多类型天生导致"同名" | 内容阅读优先于名称匹配 |
| 行数少 = 未完成 | 轻量设计文档可以是 22-54 行 | 证据链（实施产物/CHANGELOG）优先于统计 |
| 缺下游产物 = 断裂 | Plan/Spec/复盘/规则文件可互相替代 | 替代证据验证优先于结构完整性检查 |

**建议**：未来的审计工具应区分"信号"（需人工确认）和"债务"（可自动修复），避免将信号直接标记为债务。
