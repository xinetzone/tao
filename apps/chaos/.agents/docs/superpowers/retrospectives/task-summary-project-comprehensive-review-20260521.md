## 2. 目标背景

### 初始背景
随着 AgentForge 项目复杂度的增加，早期的粗放式开发暴露出多个瓶颈：
1. **平台兼容性**：核心的 `skill-creator` 评测循环在 Windows 环境下因底层 API 差异（如管道读取、编码默认值）频频崩溃。
2. **上下文污染**：AI 的内部规范、复盘报告与面向人类开发者的官方文档混杂，导致 LLM 在读取项目上下文时产生严重的幻觉（如将技能规范误认为远程 API 服务）。
3. **变更追踪缺失**：缺乏统一且模块化的 CHANGELOG，无法有效追踪各独立技能与核心基建的版本演进。

### 约束条件
- **跨平台一致性**：必须在不增加外部重量级依赖的前提下，实现 Windows/Unix 统一的脚本执行体验。
- **约定优于配置**：所有重构必须遵循现有架构风格，文档体系需平滑接入现有的 Sphinx / MyST Parser 编译流。
- **极简干预**：优先通过系统原生方案解决问题（如 PowerShell、Python 标准库）。

## 3. 执行过程

本阶段的演进分为三大核心战役：

### 战役一：技能评测链路的 Windows 兼容性攻坚 (05-20)
- **诊断**：精准定位 `select.select()` 导致 `[WinError 10038]` 及 `GBK` 编码引发的 `UnicodeDecodeError`。
- **重构**：基于 TDD（测试驱动开发）模式，编写 `test_windows_compat.py`，使用多线程+队列轮询替代了非阻塞 `select`，并全局强制声明 `encoding="utf-8"`。
- **验证**：顺利跑通 5 轮完整的 `run_loop` 自动化触发评估循环（共计 300 次调用），零报错。

### 战役二：`task-execution-summary` 技能深度调优 (05-20)
- **发现**：纯依赖自动化优化器容易导致描述失真（长句召回率低）。
- **行动**：采用“机器海选候选描述 + 人工微调收口”的混合模式，成功产出 v2.4 版 `SKILL.md`，在保证高召回的同时大幅压缩 Token 消耗。

### 战役三：全局文档架构与变更日志重塑 (05-21)
- **契约确立**：更新 `AGENTS.md`，正式划分 `docs/`（人类）与 `.agents/docs/superpowers/`（AI 技能脑区）。
- **模块化变更**：参照 `flexloop` 规范，拆分项目级与模块级 `CHANGELOG`，将旧目录迁移至 `tests/project_changelogs`，并通过 `include` 指令动态接入 Sphinx 站点。

## 5. 问题解决记录

### A: 文件 I/O 与子进程管道编码崩溃
- **现象**：`subprocess.run(text=True)` 默认采用系统区域设置 (GBK)，导致处理 Emoji 或复杂中文字符时抛出异常。
- **方案**：废弃单纯的 `text=True`，全链路显式使用 `subprocess.run(encoding="utf-8")` 及 `Path.read_text(encoding="utf-8")`。

### B: Windows 管道非阻塞读取受限
- **现象**：`run_eval.py` 在执行时因调用 Unix 专属的管道 `select()` 而全线崩溃。
- **方案**：编写 `iter_process_lines()`，通过守护线程读取 `stdout/stderr` 并推入线程安全队列，主线程轮询获取。

### C: 技能规范导致的模型幻觉
- **现象**：将技能文档写成产品 API 规范后，LLM 开始虚构复杂的远程 API 集成方案。
- **方案**：回归 Prompt 本质，精简 `SKILL.md` 描述，强调指令契约与参数映射，去除非必要的服务端暗示。

## 9. 经验方法 (方法论沉淀)

- **跨平台工程规范**：在编写面向未来的 Python 脚本时，**必须**显式指定 `encoding="utf-8"`；在处理系统级管道交互时，应避免使用依赖特定 OS 的系统调用。
- **AI 记忆与上下文治理**：将“人类说明书”与“AI 认知图谱”进行物理隔离，是大型 Agentic 项目避免幻觉、保持上下文整洁的核心最佳实践。
- **重构方法论**：在复杂项目中引入新结构时，坚持 **“分析基准模板 -> TDD 建立核心逻辑/骨架 -> 注入数据/迁移目录 -> CI集成验证”** 的闭环路径。
