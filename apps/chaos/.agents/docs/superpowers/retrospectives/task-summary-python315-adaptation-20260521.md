## 5. 问题解决

### 问题 1：defuddle 未安装
- **现象**：执行 `defuddle parse` 报 "not recognized"
- **根因**：环境未预装 defuddle npm 包
- **解决**：`npm install -g defuddle`，安装 22 个包耗时 3 秒
- **教训**：在执行外部工具调用前应先检查可用性

### 问题 2：文件过大无法一次性读取
- **现象**：`Read` 工具提示 181KB 超过 128KB 限制
- **根因**：Python What's New 文档内容极为详尽
- **解决**：分段读取（0-300, 301-600, 601-900, 901-1200），每段 300 行
- **教训**：大型文档抓取后应预判分段读取需求，避免额外 round-trip

### 问题 3：文档引用过时
- **现象**：5 份文档的"数据来源"指向已删除的临时文件
- **根因**：子智能体在生成文档时直接引用了本地临时路径
- **解决**：逐一替换为 Python 官方文档永久链接
- **教训**：子智能体生成文档时应统一规范"引用策略"，避免写入临时路径

## 9. 经验方法

### 9.1 可复用方法论

1. **"抓取→分类→并行同步→审查→汇总"五步法**：适用于任何上游版本变更的系统性适配任务
   - Step 1: 用 defuddle 抓取官方文档 → Markdown
   - Step 2: 按领域维度分类（语言特性/标准库/安全/废弃/兼容性）
   - Step 3: 并行派发子智能体同步更新各类文档
   - Step 4: 代码合规性自动化扫描
   - Step 5: 汇总生成综合报告

2. **Spec 驱动的文档任务**：即使是非代码类任务（文档更新、知识整理），使用 spec.md + tasks.md + checklist.md 的三件套仍能显著提升执行的结构化程度和可验证性。

3. **子智能体分工原则**：将"创造性写作"（文档生成）和"模式匹配扫描"（代码审查）分配给不同的子智能体，各自独立执行互不干扰。

### 9.2 对项目的直接价值

- **技术债务可视化**：tech-debt-tracker.md 按 10 条时间线整理了 100+ 条弃用项目，为后续版本升级提供清晰的行动路线图
- **代码安全确认**：确认项目未使用任何 Python 3.15 已移除/弃用的 API
- **UTF-8 一致性**：Info 级发现确认项目已有的"全链路 UTF-8"规范与 Python 3.15 的默认值一致，是正向改进

### 9.3 关于 "软弃用" 的新认知

Python 3.15 引入的 `soft deprecated` 概念是一个重要的语义变化：API 被标记为"不推荐"但**永不计划移除**（如 `re.match()` → `re.prefixmatch()`）。这类变更在传统的自动化 lint 工具中通常不会被标记，但对长期维护项目的代码风格一致性有实际影响。
