# GitHub App 安装令牌请求级覆盖头学习笔记

## 背景

2026-05-15，GitHub 为 `POST /app/installations/{installation_id}/access_tokens` 增加了临时请求头 `X-GitHub-Stateless-S2S-Token`，允许调用方在一次具体请求内显式表达“希望返回哪一类安装令牌”。`enabled` 倾向于返回新的 stateless token，`disabled` 倾向于回退到 classic stateful token，而不带该请求头时则继续遵循 GitHub 平台自身的 rollout 逻辑。这个能力的重点并不是鼓励业务长期绑定某一种 token 形态，而是给迁移期的接入方一个足够精细的观测、演练和回退开关。

## 官方要点整理

从工程实践角度看，这次能力升级最值得记住的不是“新 token 看起来像 JWT”，而是“业务代码必须继续把它当成 opaque secret”。GitHub 官方已经明确表达了几个关键信号：

- 新旧安装令牌都继续以 `ghs_` 前缀开头，不能再依靠前缀区分能力。
- 新的 stateless token 在 `ghs_` 之后通常包含两个点，看起来像三段式结构，长度显著变长。
- 旧的 stateful token 更短，通常不包含点，但这只是当前观测特征，不应被固化为业务假设。
- 请求级覆盖头属于过渡期兼容能力，未来可能移除，因此实现必须具备可回退、可拆除、可集中治理的特性。
- 该能力主要面向 GitHub.com / GHEC 等云端环境，不适用于 GHES；也就是说，环境识别与自动降级不是增强项，而是必需项。

这意味着接入方最容易踩坑的地方，并不在“如何解码 token”，而在历史系统里那些默认旧 token 形态的隐含假设，例如数据库字段长度、日志截断规则、正则校验、前端表单校验、监控脱敏模板、缓存键设计，以及各种“只要看到两个点就去解析”的调试脚本。只要这些假设还在，哪怕主流程能拿到新 token，也会在边缘链路被打断。

## 对 AgentForge 的启发

本项目采用 `config + client + cache + token_manager + CLI` 的五层设计，正是为了把这种临时能力收敛到少量、可替换的模块中。上层调用者只表达“请求意图”，例如 `auto`、`enabled` 或 `disabled`；底层编排器再结合运行环境决定是否真正发送覆盖头。这样做有三点直接收益。

第一，兼容性更稳。对于 GHES 或未知环境，系统会把请求策略降解为“不发送覆盖头”，避免把云端特性误当成通用协议，减少接入方在自建实例上的意外失败。

第二，可观测性更强。`InstallationTokenResult` 不只返回 token 本身，还携带 `requested_strategy`、`effective_strategy`、`token_kind` 与 `degraded` 等元信息。这样在调试与测试时可以区分“调用方想要什么”和“系统实际做了什么”，对于灰度迁移尤其关键。

第三，性能与稳定性更好。安装令牌本身具有短时效，而高并发场景容易出现同一个 installation 被并发刷新多次的问题。项目通过进程内缓存和单飞刷新把相同 key 的并发请求压缩为一次上游调用，在 25 并发与 100 并发模拟场景中都只触发了 1 次上游请求，这比“每个请求都去换 token”的朴素实现更接近真实生产需要。

## 接入建议

如果后续要把该能力接到更多自动化任务或工作流里，建议遵循下面几条原则。

1. 默认使用 `auto`。只有在演练新格式兼容性、做回归验证或排查线上问题时，才显式切到 `enabled` 或 `disabled`。
2. 永远不要解析 token 载荷做业务判断。就算当前 stateless token 呈现 JWT 外形，也应只做“长度、预览、类型识别”这种诊断层面的处理。
3. 日志必须脱敏。输出前缀和后缀片段足以排障，不应打印完整 token、App JWT 或私钥。
4. 在多实例部署前，不要误以为进程内缓存已经解决全部并发问题。当前实现对单进程足够有效，但跨进程、跨机器场景仍需要外部缓存和分布式协调。
5. 让覆盖头逻辑保持集中。未来如果 GitHub 下线该能力，只要调整 `client` 与 `token_manager` 的少数分支即可平滑移除，而不需要全仓搜索散落的 header 常量。

## 本次落地的指标摘要

结合当前测试结果，可以得到一组很实用的迁移指标：

- 同 key 25 并发请求：上游安装令牌请求数由理论上的 25 次压缩到 1 次。
- 同 key 100 并发压力测试：上游安装令牌请求数压缩到 1 次，失败率为 0%。
- 请求级控制能力：从“无显式覆盖能力”提升为支持 `auto / enabled / disabled` 三种意图表达。
- 诊断输出：从无统一约束提升为 CLI 默认只输出脱敏预览与结构化元数据。
- 文档沉淀：从零散实现说明提升为 README、文档站点、项目变更日志和测试报告四处同步可追踪。

综合来看，这次 GitHub App 安装令牌覆盖头的价值，不在于“拿到一种更酷的新 token”，而在于借这个迁移窗口把认证层的边界、安全约束、并发治理和文档治理一起做扎实。只要系统继续坚持 opaque secret 原则、环境自动降级和集中式策略控制，即便未来官方移除这个临时请求头，项目也能以较低成本完成二次收敛。
