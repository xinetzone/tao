# 测试域规则

本规则适用于 `tests/` 目录下的测试新增、测试修复、测试重构与测试验证任务。

## 1. 基本原则

- 测试必须服务于项目真实行为，优先验证公共接口与稳定契约。
- 测试代码应保持清晰、可维护，避免复制生产代码的内部实现细节。
- 新增测试应优先参考同目录已有测试文件的命名、断言风格与 fixture 组织方式。
- 修复测试时应先判断失败原因来自测试假设、生产代码行为变化还是环境配置问题。

## 2. 路径与导入

- 禁止在测试代码中写入本机绝对路径。
- 项目内文件引用应使用相对路径、临时目录或 pytest 提供的路径 fixture。
- 测试代码导入被测对象时，应通过包结构或公共入口导入。
- 默认禁止在测试中动态修改 `sys.path`；如确需例外，应优先检查是否可以通过包安装、测试配置或模块入口解决。

## 3. 依赖与环境

- Python 依赖与虚拟环境管理遵循根目录 `AGENTS.md`，统一使用 `uv`。
- 禁止直接使用 `pip` 或 `conda` 安装项目测试依赖。
- 不应在测试中依赖开发者本机特定工具路径、用户目录或 IDE 状态。

## 4. 测试组织

- 测试文件命名应遵循 `test_*.py`。
- 测试函数命名应描述被验证行为，而不仅描述实现细节。
- 与特定模块相关的测试应放在对应子目录中，避免把无关测试集中到单个大文件。
- 共享 fixture 应仅在复用价值明确时抽取，避免过早全局化。

## 5. 外部服务与副作用

- 默认避免真实网络请求、真实 GitHub API 调用、真实文件系统持久副作用或依赖本机全局状态。
- 对外部服务应优先使用 mock、fake client、临时目录或可控测试替身。
- 涉及并发、重试、速率限制或 token 缓存的测试，应保证结果确定、可重复。

## 6. 验证要求

- 修改测试或被测代码后，应运行与变更范围最相关的测试。
- GitHub App 专项测试优先使用 `mise run test-github-app`。
- 完整测试集使用 `mise run test`，覆盖率验证使用 `mise run test-coverage`。
- 任务完成前，如项目提供 lint、typecheck 或统一检查命令，应一并运行。
- 如无法确定统一验证命令，应先检查项目配置、README 或现有自动化脚本，再向用户确认。
